# 관리자 페이지 구현 기획안

## 1. 문서 목적
- 메인 페이지에 `Admin` 버튼을 추가하고, 별도의 관리자 페이지로 이동할 수 있도록 만드는 구현 방향을 정리합니다.
- 관리자 페이지에서는 운영자가 직접 메인 페이지의 일부 내용을 수정할 수 있도록 합니다.
- 이번 1차 범위는 복잡한 서버 개발 없이, 현재 프로젝트 구조 안에서 가장 단순하고 안전하게 적용하는 방법을 기준으로 잡습니다.

## 2. 이번 작업에서 만들 기능
### 2-1. 메인 페이지 footer에 `Admin` 버튼 추가
- 메인 페이지 footer 맨 아래에 `Admin` 텍스트 버튼을 추가합니다.
- 버튼을 누르면 관리자 페이지로 이동합니다.

### 2-2. 관리자 페이지 진입 시 비밀번호 확인
- 관리자 페이지에 들어가기 전에 비밀번호를 입력해야 합니다.
- 비밀번호는 `0000`입니다.
- 비밀번호가 맞으면 관리자 화면으로 들어갑니다.
- 비밀번호가 틀리면 진입하지 못하고 안내 문구를 보여줍니다.

### 2-3. 관리자 페이지에서 수정할 항목
- `뉴스정보(news ticker) 입력창`
- `판매 수치(sales counter) 재설정 입력창`
- `제품 판매 빈도 설정창(Dot_blue 발생 빈도)`
- `알림 수신 메일 주소 입력창`

### 2-4. 관리자 페이지 상단 로고 동작
- 좌측 상단의 HAGOBOGO 로고를 누르면 메인 화면으로 이동합니다.

## 3. 왜 이렇게 구현하는가
현재 프로젝트는 React Router 같은 라우팅 도구 없이 메인 화면 하나를 중심으로 동작하는 구조입니다.

이 상태에서 가장 안전한 방법은 아래와 같습니다.

### 3-1. 페이지 구분 방식
- 메인 페이지: `app.html`
- 관리자 페이지: `app.html?view=admin`

이 방식의 장점:
- 구조를 크게 바꾸지 않아도 됩니다.
- 새로운 라이브러리를 추가하지 않아도 됩니다.
- 유지보수가 쉽습니다.

## 4. 관리자 비밀번호 처리 방식
### 4-1. 처리 흐름
1. 사용자가 메인 footer의 `Admin` 버튼을 누릅니다.
2. `app.html?view=admin`으로 이동합니다.
3. 관리자 인증 상태가 없으면 비밀번호 입력 화면이 먼저 보입니다.
4. 비밀번호 `0000`을 입력하면 관리자 페이지로 들어갑니다.
5. 인증 성공 상태는 세션 저장소에 잠시 저장합니다.

### 4-2. 저장 방식
- `sessionStorage`를 사용합니다.
- 브라우저 탭을 닫으면 인증 상태는 사라집니다.

### 4-3. 중요한 한계
- 이 방식은 프론트엔드만으로 처리하는 간단한 접근 제한입니다.
- 비밀번호가 코드 안에 들어가므로 강한 보안 방식은 아닙니다.
- 즉, “운영 편의용 잠금”에는 적합하지만 “민감한 관리자 보안”에는 적합하지 않습니다.

## 5. 뉴스 티커 수정 기능 기획
### 5-1. 목표
- 메인 상단에 흘러가는 `news ticker` 문구를 관리자가 직접 바꿀 수 있어야 합니다.

### 5-2. 입력 방식
- 관리자 페이지에 여러 줄 텍스트 입력창을 둡니다.
- 한 줄에 한 문장씩 입력합니다.
- 저장하면 메인 페이지의 ticker가 같은 순서로 반영됩니다.

### 5-3. 저장 방식
- `localStorage`에 배열 형태로 저장합니다.
- 저장된 값이 있으면 메인 페이지는 기본 번역 문구 대신 관리자 입력값을 우선 사용합니다.

### 5-4. 장점
- 서버 없이 바로 적용됩니다.
- 새로고침 후에도 값이 유지됩니다.
- 운영자가 개발자 도움 없이 문구를 바꿀 수 있습니다.

## 6. Dot_blue 발생 빈도 설정 기능 기획
### 6-1. 목표
- 메인 페이지의 `Dot_blue`가 얼마나 자주 발생할지 관리자가 직접 조정할 수 있어야 합니다.

### 6-2. 입력 방식
- 관리자 페이지에 숫자 입력창을 둡니다.
- 단위는 `초`로 표시합니다.
- 예: `2`를 입력하면 2초마다 한 번 정도 발생하도록 설정합니다.

### 6-3. 저장 방식
- `localStorage`에 숫자 값으로 저장합니다.
- 메인 페이지의 `DotBlueEngine`이 이 값을 읽어서 생성 간격으로 사용합니다.

### 6-4. 왜 초 단위가 좋은가
- 초보자도 이해하기 쉽습니다.
- `1500ms`, `2400ms` 같은 개발자용 숫자보다 운영자가 다루기 편합니다.

## 7. 알림 수신 메일 주소 변경 기능 기획
### 7-1. 목표
- 문의사항이 접수되었을 때 알림을 받는 메일 주소를 관리자 페이지에서 직접 바꿀 수 있어야 합니다.

### 7-2. 왜 프론트만으로는 해결되지 않는가
- 현재 문의 데이터는 프론트 화면에서 바로 메일을 보내는 구조가 아닙니다.
- 프론트는 문의 내용을 `Google Apps Script`로 전달하고,
- 실제로 메일을 보내는 쪽은 `Apps Script`입니다.

즉, 메일 수신 주소는 브라우저 화면이 아니라 `Apps Script` 쪽 설정으로 관리해야 합니다.

### 7-3. 추천 구현 방식
1. 관리자 페이지에 `알림 수신 메일 주소` 입력창을 추가합니다.
2. 저장 버튼을 누르면 프론트가 `Apps Script 설정용 API`를 호출합니다.
3. `Apps Script`가 전달받은 메일 주소 형식을 검증합니다.
4. 검증이 통과하면 `Script Properties` 또는 별도 설정 저장 위치에 메일 주소를 저장합니다.
5. 이후 문의 메일 발송 시 저장된 메일 주소를 읽어서 사용합니다.

### 7-4. 저장 위치 추천
가장 추천하는 방식은 `Apps Script Script Properties`에 저장하는 것입니다.

이유:
- 설정값과 운영 코드를 같은 영역에서 관리할 수 있습니다.
- Google Sheets 본문 데이터와 운영 설정을 분리할 수 있습니다.
- 문의 데이터 표를 불필요하게 복잡하게 만들지 않아도 됩니다.

### 7-5. 검증 규칙
- 메일 주소는 비어 있으면 안 됩니다.
- 이메일 형식 검사 정규식을 통과해야 합니다.
- 필요하면 허용 도메인을 제한할 수도 있습니다.

### 7-6. 보안상 주의할 점
- 프론트가 보낸 메일 주소를 아무 검증 없이 바로 저장하면 안 됩니다.
- 관리자 인증 없이 누구나 설정 API를 호출할 수 있으면 안 됩니다.
- 최소한 관리자 비밀번호 또는 별도 토큰 검증이 필요합니다.

### 7-7. 추천 보안 흐름
1. Admin 페이지에서 비밀번호 인증
2. 메일 주소 저장 요청 시 인증 토큰 또는 비밀번호 함께 전달
3. Apps Script가 인증값을 확인
4. 통과한 경우에만 메일 주소 저장

### 7-8. 문의 메일 발송 흐름 변경점
기존:
- 문의 데이터 저장
- 고정된 메일 주소로 발송

변경 후:
- 문의 데이터 저장
- `Script Properties` 또는 설정 저장소에서 현재 메일 주소 조회
- 조회된 주소로 발송

## 8. 판매 수치(sales counter) 재설정 기능 기획
### 8-1. 목표
- 관리자 페이지에서 현재 판매 수치를 직접 입력해 메인 페이지의 `sales counter` 시작값을 바꿀 수 있어야 합니다.

### 8-2. 기능 정의
- 관리자 페이지에 숫자 입력창을 추가합니다.
- 운영자가 새 판매 수치를 입력하고 저장하면 메인 페이지의 카운터 기준값이 그 숫자로 바뀝니다.
- 이후 `dot_blue` 충돌이 발생하면 그 기준값에서 다시 증가합니다.

### 8-3. 추천 동작 방식
이번 1차 구현에서는 `고정값 유지`가 아니라 `초기값 재설정` 방식으로 가는 것이 맞습니다.

의미:
- 저장 시점에 카운터의 현재 기준값을 새 숫자로 바꿉니다.
- 그 뒤에는 기존처럼 충돌할 때마다 숫자가 계속 증가합니다.

이 방식을 추천하는 이유:
- 지금 메인 페이지는 `dot_blue` 충돌로 판매 카운터가 계속 올라가는 구조입니다.
- 만약 관리자 입력값을 항상 강제로 덮어쓰는 방식으로 만들면, 충돌로 증가한 값이 바로 다시 사라져 사용자가 보기 어색해집니다.

### 8-4. 저장 방식
- `localStorage`를 사용합니다.
- 메인 페이지가 처음 열릴 때, 관리자 페이지에서 저장한 판매 기준값이 있으면 그 값을 우선 사용합니다.
- 값이 없으면 기존 기본값 `100000` 또는 기존 저장값을 사용합니다.

### 8-5. 우선순위 제안
카운터 값은 아래 우선순위로 읽는 것이 가장 안전합니다.

1. 관리자 페이지에서 최근 저장한 판매 기준값
2. 기존 사용 중이던 `sphere_count`
3. 기본값 `100000`

이렇게 하면 운영자가 새 기준값을 저장했을 때 메인 화면에 바로 반영되고, 이후 증가한 값도 자연스럽게 이어집니다.

### 8-6. 입력 규칙
- 숫자만 입력 가능
- 0 이상만 허용
- 소수점 없이 정수 기준 사용 권장
- 너무 큰 값이 들어오는 경우를 막기 위해 상한선을 둘 수도 있습니다

### 8-7. 화면 구성
- 관리자 페이지에 `판매 수치(sales counter) 재설정` 섹션 추가
- 현재 저장된 값 표시
- 새 숫자 입력창
- 저장 버튼
- 저장 완료 메시지

### 8-8. 데이터 흐름
1. 운영자가 관리자 페이지에서 새 판매 수치 입력
2. 저장 버튼 클릭
3. `localStorage`에 판매 기준값 저장
4. 메인 페이지 진입 또는 재반영 시 저장값 읽기
5. `sales` 상태의 시작값으로 적용
6. 이후 `dot_blue` 충돌 시 기존처럼 `+1`씩 증가

## 15. 최종 판단
이번 관리자 페이지는 “운영자가 직접 문구와 Dot_blue 발생 빈도를 수정할 수 있게 하는 간단한 운영 도구”로 보는 것이 맞습니다.

현재 프로젝트 구조와 난이도를 고려하면,
- `query string` 기반 페이지 분기
- `sessionStorage` 기반 간단한 비밀번호 진입 제한
- `localStorage` 기반 설정 저장
- `Apps Script` 기반 알림 메일 주소 저장

이 3가지를 조합하는 방식이 가장 현실적이고 유지보수도 쉽습니다.

다만 알림 메일 주소 변경은 프론트만 수정해서 끝나는 기능이 아니므로,
- Admin UI
- Apps Script 설정 저장 API
- Apps Script 메일 발송 로직

이 세 부분을 같이 봐야 안정적으로 동작합니다.
