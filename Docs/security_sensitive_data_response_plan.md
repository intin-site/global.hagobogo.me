# 민감정보 노출 대응 정리

## 1. 이 문서의 목적

이 문서는 문의 저장 기능 작업 중 저장소와 빌드 결과물에 포함되었을 수 있는
민감정보와 준민감정보를 어떻게 정리하고 대응할지 정리한 문서입니다.

여기서 말하는 대상은 아래와 같습니다.

- Apps Script 웹앱 `/exec` URL
- 프론트에서 사용하는 `requestToken`
- `.env.local`, `.env.production` 같은 환경 파일
- 이미 원격 저장소에 올라간 과거 커밋 이력

## 2. 먼저 분명히 알아야 할 점

`VITE_`로 시작하는 값은 프론트 빌드에 포함됩니다.

즉:

- `VITE_BUSINESS_INQUIRY_API_URL`
- `VITE_BUSINESS_INQUIRY_REQUEST_TOKEN`

이 두 값은 브라우저에서 최종적으로 확인될 수 있는 값입니다.

따라서 이 값은 서버 비밀키처럼 강한 비밀정보로 취급하면 안 됩니다.

특히 `requestToken`은 “완전한 보안 장치”가 아니라
“무분별한 요청을 조금 줄이는 1차 필터” 정도로 이해하는 것이 맞습니다.

## 3. 이번 정리에서 이미 반영한 것

현재 저장소 기준으로 아래 조치를 반영했습니다.

1. 실제 값이 들어 있던 `.env.local`, `.env.production`을 예시값으로 교체
2. `.gitignore`에 `.env.*` 패턴을 추가하고 예시 파일만 추적 유지
3. Apps Script 코드가 더 이상 코드 안에 실제 토큰을 하드코딩하지 않고
   `Script Properties`에서 읽도록 변경
4. 예시 문서 안의 실제 토큰 문자열도 `REPLACE_WITH_NEW_REQUEST_TOKEN` 형태로 교체

즉, **현재 작업 트리 기준으로는 더 이상 실제 값이 남지 않도록 정리하는 방향**으로 바뀌었습니다.

## 4. 대장님이 바로 해야 하는 실제 조치

### 4-1. Apps Script 토큰 교체

가장 먼저 새 토큰을 하나 다시 정해야 합니다.

예:

```text
REPLACE_WITH_NEW_REQUEST_TOKEN
```

실전에서는 이 값을 더 길고 추측하기 어렵게 만들어야 합니다.

그 다음 Apps Script 편집기에서 아래 순서로 설정합니다.

1. `프로젝트 설정`으로 이동
2. `스크립트 속성` 열기
3. 아래 키 추가

```text
키: BUSINESS_INQUIRY_REQUEST_TOKEN
값: 새로 만든 실제 토큰
```

4. 웹앱 다시 배포

### 4-2. 로컬 환경 파일에 새 값 입력

로컬에서는 아래 파일 중 필요한 곳에 새 값을 다시 넣어야 합니다.

- `.env.local`
- `.env.production`

예:

```env
VITE_BUSINESS_INQUIRY_API_URL=새_Apps_Script_exec_URL
VITE_BUSINESS_INQUIRY_REQUEST_TOKEN=새로_만든_실제_토큰
```

중요:

- 이 파일들은 이제 Git 추적 대상이 아니어야 합니다.
- 즉, 로컬에서만 사용하고 커밋하지 않는 것이 원칙입니다.

### 4-3. 프론트 재빌드

새 값 입력 후에는 다시 빌드해야 합니다.

```bash
npm run build
```

## 5. 이미 푸시된 커밋 이력은 어떻게 볼 것인가

이미 원격 저장소에 올라간 커밋에는 과거 값이 남아 있을 수 있습니다.

이 경우 대응 수준은 두 가지로 나뉩니다.

### 대응 A. 빠른 현실 대응

이 방식은 가장 먼저 해야 하는 최소 대응입니다.

1. 토큰 교체
2. Apps Script 재배포
3. 로컬 환경 파일 갱신
4. 새 빌드 반영

장점:

- 빠릅니다.
- 운영 복구가 쉽습니다.

한계:

- 과거 커밋 기록 자체는 그대로 남습니다.

### 대응 B. 이력 정리까지 포함한 대응

저장소가 공개 저장소이거나,
과거 민감값 노출 흔적 자체를 줄이고 싶다면 이력 정리까지 검토해야 합니다.

가능한 방식:

1. Git 히스토리 재작성
2. 민감 문자열이 포함된 커밋 제거
3. 강제 푸시
4. 협업자 로컬 저장소도 다시 맞추기

주의:

- 이 작업은 기존 커밋 해시가 전부 바뀔 수 있습니다.
- 협업 중이면 충돌이 생기기 쉽습니다.
- 잘못하면 작업 이력이 꼬일 수 있으니 신중하게 해야 합니다.

즉, **대응 B는 보안상 더 깔끔하지만 운영 리스크가 더 큽니다.**

## 6. 추천 판단

현재 상황에서는 아래 순서를 권장합니다.

1. 새 토큰으로 즉시 교체
2. Apps Script 재배포
3. 로컬 환경 파일 갱신
4. 새 빌드 반영
5. 저장소 공개 범위와 협업 상황을 확인한 뒤 이력 정리 여부 결정

즉, 먼저 운영상 위험을 끊고,
그 다음 Git 이력 정리는 별도 결정으로 가져가는 편이 안전합니다.

## 7. 장기적으로 더 안전한 구조

현재 구조는 정적 프론트가 Apps Script를 직접 호출합니다.

이 구조의 한계:

- 프론트 값은 결국 브라우저에 노출됩니다.
- 따라서 `requestToken`은 강한 비밀값 역할을 못 합니다.

더 안전하게 가려면 아래 구조가 필요합니다.

`프론트 -> 서버 중계 API -> Apps Script 또는 Google Sheets`

이렇게 하면 토큰이나 키를 프론트에 두지 않아도 됩니다.

즉, 정말 비밀로 지켜야 하는 값이 있다면
장기적으로는 서버 중계 구조로 옮기는 것이 맞습니다.
