# 판매 카운트 최신값 동기화 기획안

## 1. 문제 요약
- 현재 화면에서 보이는 판매 카운트와 새 브라우저에서 처음 보이는 판매 카운트가 서로 다릅니다.
- 예를 들어 현재 어떤 브라우저에서는 `108,524`가 보이는데, 새 브라우저로 접속하면 관리자 설정에 저장된 `108,507`이 먼저 보입니다.

## 2. 왜 이런 일이 생기는가
### 현재 코드 구조
- [`src/components/Dashboard.jsx`](/Users/chris/development/AntigravityWorks/site_global.hagobogo/src/components/Dashboard.jsx)에서 서버 설정값을 `baseSales`로 읽습니다.
- 같은 파일에서 파란 점이 충돌할 때마다 `localHits`를 `1`씩 올립니다.
- 실제 화면에는 `displayedSales = baseSales + localHits` 값이 표시됩니다.

### 초보자용 쉬운 설명
- `baseSales`는 서버에 저장된 공식 숫자입니다.
- `localHits`는 지금 보고 있는 브라우저 안에서만 임시로 더해지는 숫자입니다.
- 그래서 지금 브라우저에서는 숫자가 계속 올라가지만, 다른 브라우저는 그 임시 증가분을 모르기 때문에 서버에 저장된 예전 숫자만 보이게 됩니다.

## 3. 핵심 원인
- 판매 카운트 증가분이 서버에 저장되지 않습니다.
- 증가분이 브라우저 메모리 상태에만 존재합니다.
- 따라서 브라우저를 바꾸거나 새로 열면 증가분이 사라집니다.

## 4. 지금 구조의 한계
### 한계 1. 브라우저마다 숫자가 달라질 수 있음
- 같은 시점이어도 각 브라우저가 따로 증가분을 계산하므로 화면 숫자가 서로 다를 수 있습니다.

### 한계 2. 새 브라우저는 항상 공식 기준값부터 시작함
- 서버에는 `salesCount`만 있고 최근 증가 이력은 없으므로 새 브라우저는 최신 화면 값을 재현할 수 없습니다.

### 한계 3. 시간이 지날수록 실제 숫자 의미가 불명확해짐
- 현재 숫자가 "운영 기준 공식 누적값"인지, "현재 브라우저에서 잠깐 연출된 숫자"인지 구분이 흐려집니다.

## 5. 해결 방향 후보
### 안 A. 브라우저 연출용 증가분을 서버에 즉시 누적 저장
#### 방식
- 파란 점 충돌이 발생할 때마다 서버에 `salesCount + 1` 저장 요청을 보냅니다.
- 모든 브라우저는 일정 주기 또는 진입 시 서버 최신값을 다시 읽습니다.

#### 장점
- 구조가 가장 단순합니다.
- 새 브라우저도 거의 최신값을 볼 수 있습니다.

#### 단점
- 충돌이 자주 발생하면 서버 저장 요청이 너무 많아집니다.
- 동시에 여러 브라우저가 요청하면 값이 꼬일 수 있어 서버 쪽 원자적 증가 처리 로직이 꼭 필요합니다.
- Apps Script는 짧은 간격의 빈번한 쓰기 요청에 불리할 수 있습니다.

### 안 B. 서버에 "공식 누적값"만 저장하고, 브라우저는 연출 전용 증가분만 잠깐 보여준 뒤 주기적으로 서버값에 다시 맞춤
#### 방식
- 지금처럼 화면에서는 부드럽게 증가 애니메이션을 보여줍니다.
- 하지만 일정 주기마다 서버 공식값을 다시 읽어 화면을 서버 기준으로 보정합니다.

#### 장점
- 구현 난도가 낮습니다.
- 서버 부하가 적습니다.

#### 단점
- 새 브라우저와 기존 브라우저 숫자가 완전히 같아지지 않을 수 있습니다.
- 대장님이 원하는 "최신 값 싱크" 요구에는 부족할 가능성이 큽니다.

### 안 C. 서버에 "마지막 공식 카운트"와 "증가 이벤트 누적치"를 함께 저장
#### 방식
- 관리자 설정의 `salesCount`는 초기 기준값으로 유지합니다.
- 별도로 서버에 `salesIncrement` 또는 `lastSyncedSalesCount` 같은 값을 둡니다.
- 브라우저 충돌 이벤트는 바로 저장하지 않고 짧은 주기로 묶어서 서버에 반영합니다.
- 새 브라우저는 `공식 기준값 + 누적 증가분`을 받아 바로 최신값에 가깝게 시작합니다.

#### 장점
- 관리자 기준값과 실시간 누적값 역할을 분리할 수 있습니다.
- 빈번한 충돌 요청을 묶어 저장하면 서버 부담을 줄일 수 있습니다.
- 새 브라우저도 최신값에 더 가깝게 맞출 수 있습니다.

#### 단점
- 프론트와 Apps Script를 둘 다 수정해야 합니다.
- 배치 저장 주기와 충돌 처리 규칙을 정해야 합니다.

## 6. 추천안
### 추천: 안 C
- 현재 프로젝트 구조에서는 가장 현실적입니다.
- 이유는 아래와 같습니다.

1. 관리자 페이지의 `salesCount`는 "운영자가 직접 정하는 시작 기준값" 역할로 유지할 수 있습니다.
2. 실시간 증가분은 별도 누적값으로 관리해 책임이 분리됩니다.
3. 충돌마다 저장하지 않고 `2초~5초 간격` 또는 `N건 누적 시` 묶어서 저장하면 Apps Script 부담을 줄일 수 있습니다.
4. 새 브라우저는 진입 즉시 최신 누적값을 읽어 기존 브라우저와 큰 차이 없이 시작할 수 있습니다.

## 7. 권장 데이터 구조
### Apps Script Script Properties 예시
- `SITE_SALES_COUNT`
  - 관리자 기준 시작값
- `SITE_SALES_INCREMENT`
  - 기준값 이후 누적 증가분
- `SITE_SETTINGS_UPDATED_AT`
  - 마지막 설정 변경 시각
- `SITE_SALES_UPDATED_AT`
  - 마지막 증가분 반영 시각

### 화면 계산 공식
- 최종 표시값 = `SITE_SALES_COUNT + SITE_SALES_INCREMENT + 현재 브라우저에서 아직 서버에 보내지 않은 임시 증가분`

## 8. 권장 동작 순서
### 메인 페이지
1. 페이지 진입 시 서버에서 `기준값`과 `누적 증가분`을 함께 읽습니다.
2. 화면은 두 값을 더한 숫자에서 시작합니다.
3. 파란 점 충돌이 발생하면 화면 숫자는 즉시 올라갑니다.
4. 증가분은 메모리에 잠깐 쌓아 두었다가 짧은 주기로 서버에 묶어서 보냅니다.
5. 서버 반영이 끝나면 임시 증가분 버퍼를 비웁니다.
6. 주기적으로 서버 최신값을 다시 읽어 드리프트를 보정합니다.

### 관리자 페이지
1. `판매 수치` 저장 시 기준값만 바꿀지
2. 기준값과 누적 증가분을 합친 "현재 공식값"으로 저장할지
3. 위 두 방식 중 하나를 명확히 정해야 합니다.

### 추천 관리자 저장 규칙
- 관리자 화면에서 판매 수치를 저장할 때는 `현재 공식값을 새 기준값으로 저장`하고 `누적 증가분은 0으로 초기화`하는 방식이 가장 단순합니다.
- 이렇게 하면 운영자가 숫자를 수동 보정할 때 계산이 꼬이지 않습니다.

## 9. 서버 쪽 필수 조건
- 증가분 저장은 반드시 잠금 기반으로 처리해야 합니다.
- 현재 Apps Script에는 `LockService`가 이미 있으므로 이를 재사용하면 됩니다.
- 중요한 점은 "현재 값 읽기 → 더하기 → 다시 저장" 전체가 한 잠금 안에서 처리되어야 한다는 점입니다.

## 10. 구현 시 주의점
### 주의점 1. 너무 자주 저장하면 안 됨
- 파란 점 충돌마다 바로 저장하면 요청 수가 많아집니다.
- 배치 저장이 필요합니다.

### 주의점 2. 탭 여러 개가 동시에 열릴 수 있음
- 같은 사용자가 탭 여러 개를 띄우면 임시 증가분이 중복 반영될 수 있습니다.
- 브라우저 식별자 또는 탭별 버퍼 정책을 단순하게라도 정해야 합니다.

### 주의점 3. 실패 복구가 있어야 함
- 서버 저장 실패 시 임시 증가분을 버리면 숫자가 줄어든 것처럼 보일 수 있습니다.
- 실패 시 재시도 큐를 유지하는 편이 안전합니다.

## 11. 추천 구현 순서
1. Apps Script에 `누적 증가분 조회/증가` API를 추가합니다.
2. 공개 설정 조회 응답에 `현재 공식 표시값` 또는 `누적 증가분`을 포함시킵니다.
3. 프론트 `Dashboard`에서 `baseSales + localHits` 단일 구조를 `서버 공식값 + 미전송 버퍼` 구조로 바꿉니다.
4. 일정 주기 배치 저장과 실패 재시도 로직을 추가합니다.
5. 관리자 저장 시 기준값/누적값 초기화 규칙을 반영합니다.
6. 새 브라우저, 기존 브라우저, 여러 탭 동시 실행 검수를 진행합니다.

## 12. 검수 시나리오
1. 브라우저 A에서 숫자가 증가한 뒤 브라우저 B를 새로 열었을 때 거의 같은 숫자로 시작하는지 확인
2. 브라우저 A와 B를 동시에 열고 일정 시간 뒤 서로 숫자 차이가 과도하게 벌어지지 않는지 확인
3. 관리자 페이지에서 판매 수치를 강제로 수정한 뒤 모든 브라우저가 새 기준값으로 맞춰지는지 확인
4. 네트워크 일시 실패 후에도 숫자 누적이 유실되지 않는지 확인

## 13. 최종 판단
- 현재 문제는 단순 표시 버그가 아니라 `카운트의 진실 원본이 어디인가`가 아직 분리되지 않은 구조 문제입니다.
- "모든 브라우저에서 최신값을 비슷하게라도 맞추고 싶다"가 목표라면, 판매 카운트를 서버 누적형 구조로 바꾸는 것이 맞습니다.
- 지금 코드 기준으로는 `관리자 설정값`만 서버 기준이고 `증가분`은 브라우저 임시값이기 때문에, 새 브라우저와 기존 브라우저 숫자가 다르게 보이는 현상은 자연스럽게 발생합니다.
