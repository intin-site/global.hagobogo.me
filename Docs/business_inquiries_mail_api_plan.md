# Business Inquiries 메일 발송 API 구조 설계서

## 1. 문서 목적

이 문서는 `Business Inquiries` 팝업에 입력된 내용을 특정 이메일 주소로 발송하기 위한 백엔드 API 구조를 설계한 문서입니다.

목표는 단순합니다.

- 프론트엔드 팝업에서 입력값을 수집한다.
- 백엔드 API로 데이터를 전송한다.
- 백엔드가 지정된 이메일 주소로 메일을 발송한다.
- 사용자에게는 성공 또는 실패 결과만 보여준다.

## 2. 권장 방식

가장 권장되는 방식은 아래 구조입니다.

`프론트엔드 팝업 -> 백엔드 API -> 메일 발송 서비스 -> 지정 이메일 주소`

이 방식이 적절한 이유는 아래와 같습니다.

- 메일 발송용 비밀키를 브라우저에 노출하지 않아도 된다.
- 수신 주소, 발신 설정, 검증 로직을 서버에서 통제할 수 있다.
- 스팸 방지, 로그 기록, 오류 처리 구조를 확장하기 쉽다.
- `mailto:` 방식처럼 사용자 환경에 의존하지 않는다.

## 3. 전체 동작 흐름

1. 사용자가 `Business Inquiries` 팝업에 내용을 입력한다.
2. 사용자가 `Save`를 누른다.
3. 프론트엔드가 `POST /api/business-inquiry` 요청을 보낸다.
4. 백엔드가 요청 본문을 검증한다.
5. 검증 통과 시 메일 발송 서비스를 호출한다.
6. 지정된 이메일 주소로 문의 메일이 발송된다.
7. 백엔드가 성공 또는 실패 응답을 프론트엔드에 반환한다.
8. 프론트엔드는 사용자에게 완료 또는 오류 메시지를 보여준다.

## 4. 프론트엔드 역할

프론트엔드는 아래 역할만 담당하는 것이 적절합니다.

- 입력값 수집
- 기본 입력 검증
- 제출 중 상태 표시
- 중복 제출 방지
- 성공/실패 메시지 표시
- 언어 정보 포함 전송

프론트에서 다루는 데이터 예시는 아래와 같습니다.

- `name`
- `jobTitle`
- `companyName`
- `email`
- `inquiry`
- `language`
- `submittedAt`

## 5. 백엔드 역할

백엔드는 아래 역할을 담당해야 합니다.

- API 요청 수신
- 필수 입력값 검증
- 이메일 형식 검증
- 글자 수 및 길이 제한 검증
- 스팸 방지 처리
- 메일 제목/본문 생성
- 메일 서비스 호출
- 성공/실패 응답 반환

즉, 메일 발송의 실제 책임은 프론트가 아니라 서버에 있어야 합니다.

## 6. API 명세 제안

### 6-1. 엔드포인트

- `POST /api/business-inquiry`

### 6-2. 요청 본문 예시

```json
{
  "name": "홍길동",
  "jobTitle": "Marketing Manager",
  "companyName": "ABC Inc.",
  "email": "hello@example.com",
  "inquiry": "문의 내용",
  "language": "EN"
}
```

### 6-3. 성공 응답 예시

```json
{
  "ok": true,
  "message": "Inquiry submitted successfully."
}
```

### 6-4. 실패 응답 예시

```json
{
  "ok": false,
  "message": "Invalid email format."
}
```

## 7. 서버 검증 기준

최소한 아래 검증은 서버에서 다시 해야 합니다.

- `name`: 필수
- `jobTitle`: 선택 또는 정책에 따라 필수
- `companyName`: 필수
- `email`: 필수
- `email` 형식 검사
- `inquiry`: 필수
- `inquiry` 최대 길이 검사

프론트에서 이미 검증해도, 서버에서 다시 검증해야 안전합니다.

## 8. 메일 제목과 본문 설계

### 8-1. 메일 제목 예시

- `[HAGOBOGO] New Business Inquiry from ABC Inc.`

### 8-2. 메일 본문 포함 정보

- Name
- Job Title
- Company Name
- Email
- Language
- Inquiry
- Submitted At

초기 구현에서는 복잡한 템플릿보다 plain text 또는 단순 HTML이 적절합니다.

## 9. 수신 이메일 주소 관리

수신 이메일 주소는 프론트 코드에 두면 안 됩니다.  
반드시 서버 환경 변수로 관리해야 합니다.

예시:

- `BUSINESS_INQUIRY_TO_EMAIL=contact@example.com`

이렇게 해야:

- 주소 변경 시 코드 수정 없이 운영 설정만 바꾸면 된다.
- 민감한 정보가 저장소에 남지 않는다.

## 10. 메일 발송 서비스 선택

권장 우선순위는 아래와 같습니다.

1. `Resend`
2. `SendGrid`
3. `AWS SES`

현재 단계에서는 `Resend`가 가장 적절합니다.

이유:

- API가 단순하다.
- 서버 또는 서버리스에서 붙이기 쉽다.
- 초기 구현 속도가 빠르다.

## 11. 보안 설계

메일 발송 기능은 외부 입력을 받기 때문에 반드시 최소 보안 구조가 필요합니다.

최소 권장 항목:

- 서버 측 필수값 검증
- 이메일 형식 검증
- 글자 수 제한
- 요청당 rate limiting
- 환경 변수로 메일 키 관리

초기에는 CAPTCHA까지 바로 넣지 않더라도, 요청 제한은 고려하는 편이 좋습니다.

## 12. 오류 처리 설계

프론트엔드는 아래 경우를 구분해서 보여줄 수 있어야 합니다.

- 입력값 오류
- 서버 오류
- 메일 발송 서비스 오류
- 네트워크 오류

사용자 메시지는 최대한 단순하게 유지하는 것이 좋습니다.

예시:

- 성공: `문의가 정상적으로 접수되었습니다.`
- 실패: `전송에 실패했습니다. 잠시 후 다시 시도해 주세요.`

세부 오류 로그는 서버에서만 기록하는 것이 적절합니다.

## 13. 권장 구현 형태

현재 프로젝트 구조 기준으로 가장 현실적인 방식은 아래와 같습니다.

- 프론트엔드: 현재 Vite/React 구조 유지
- 백엔드: 별도 경량 API 또는 서버리스 함수
- 메일 발송: Resend 사용

추천 구조:

1. 프론트에서 `/api/business-inquiry` 호출
2. 서버리스 함수 또는 백엔드 API가 요청 수신
3. 메일 서비스 호출
4. 결과를 프론트에 반환

## 14. 단계별 구현 계획

### 1단계. API 스펙 확정

- 요청 필드 확정
- 필수 항목 확정
- 응답 형식 확정

### 2단계. 프론트 제출 로직 연결

- `Save` 클릭 시 API 호출
- 제출 중 상태 표시
- 성공/실패 메시지 표시

### 3단계. 백엔드 엔드포인트 구현

- `POST /api/business-inquiry` 생성
- 입력 검증 추가

### 4단계. 메일 서비스 연결

- Resend 또는 다른 서비스 연동
- 수신 주소 환경 변수 연결

### 5단계. 운영 안정화

- rate limiting
- 에러 로깅
- 실패 응답 정리

## 15. 비판적으로 봐야 할 점

현재 프로젝트는 정적 프론트 중심입니다.  
따라서 메일 발송 기능을 넣으려면 단순 UI 수정이 아니라, 백엔드 또는 서버리스 기능 추가가 반드시 필요합니다.

즉, 이 기능은 다음을 포함하는 구조 변경입니다.

- API 계층 추가
- 환경 변수 관리
- 배포 방식 조정 가능성

프론트 코드만 수정해서 안전한 메일 발송 기능을 만드는 것은 적절하지 않습니다.

## 16. 최종 권장안

현재 단계에서 가장 권장되는 1차 구조는 아래와 같습니다.

- 현재 `Business Inquiries` 팝업 UI 유지
- `Save` 클릭 시 백엔드 API 호출
- API는 서버리스 함수 1개로 시작
- 메일 발송은 Resend 사용
- 수신 이메일 주소와 메일 키는 환경 변수로 관리

이 구조가 구현 속도, 보안, 유지보수성 측면에서 가장 균형이 좋습니다.
