# HAGOBOGO FAQ형 챗봇 도입 기획안

## 1. 문서 목적

이 문서는 하고보고 사이트에 `FAQ형 챗봇`을 도입하기 위한 1차 기획안입니다.

이번 기획의 목표는 아래와 같습니다.

- 현재 화면 우하단의 챗봇 버튼 UI를 그대로 활용한다.
- 버튼을 누르면 작은 채팅 패널이 열리도록 만든다.
- 사용자가 자주 묻는 질문을 빠르게 확인할 수 있게 한다.
- 마지막에는 `Business Inquiries`로 자연스럽게 연결되게 만든다.

즉, 이번 챗봇은 “진짜 AI가 자유롭게 대화하는 챗봇”이 아니라,
사이트 방문자가 궁금한 내용을 빠르게 확인하고 문의까지 이어지게 만드는
`가벼운 안내형 챗봇`이 핵심입니다.

## 2. 왜 이 방식이 맞는가

대장님 사이트는 지금 단계에서 “정확한 정보 전달”과 “문의 전환”이 더 중요합니다.

이 점을 기준으로 보면, 처음부터 AI 챗봇을 붙이는 방식에는 아래 위험이 있습니다.

- 답변이 틀릴 수 있다.
- 의료/난임 관련 문맥에서 과장되거나 부정확한 표현이 나올 수 있다.
- API 비용과 서버 구성이 추가된다.
- GitHub Pages만으로는 안전하게 운영하기 어렵다.

반대로 FAQ형 챗봇은 아래 장점이 있습니다.

- 답변 내용을 대장님이 직접 통제할 수 있다.
- 구현 난이도가 낮다.
- 운영 비용이 거의 없다.
- 지금 있는 사이트 구조를 크게 흔들지 않는다.
- 나중에 AI 챗봇으로 확장할 때도 출발점이 된다.

즉, 이번 1차 도입은 “상담 전환용 스마트 메뉴”에 가깝게 보는 것이 맞습니다.

## 3. 이번 챗봇의 역할

이번 챗봇은 아래 3가지 역할만 정확히 하면 됩니다.

1. 방문자가 궁금해할 핵심 질문을 바로 보여준다.
2. 제안서, 제품 특징, 문의 흐름을 짧고 명확하게 안내한다.
3. 더 깊은 상담이 필요하면 `Business Inquiries`로 넘긴다.

이 역할을 벗어나 너무 많은 것을 하려고 하면 오히려 복잡해집니다.

예를 들어 이번 1차 버전에서는 아래 기능은 넣지 않는 것이 좋습니다.

- 자유 입력 AI 대화
- 로그인 기반 개인화
- 실시간 상담원 연결
- 복잡한 추천 엔진

## 4. 화면 구조 제안

### 4-1. 시작점

현재 우하단에 있는 챗봇 버튼을 그대로 사용합니다.

즉, 사용자가 이미 보고 있는 원형 버튼은 유지하고
클릭했을 때만 작은 채팅 패널이 열리는 방식으로 갑니다.

### 4-2. 열리는 패널 형태

챗봇 패널은 아래 성격이 적절합니다.

- 위치: 우하단 버튼 바로 위
- 크기: 데스크톱 기준 중소형 패널
- 형태: 살짝 둥근 카드
- 배경: 현재 사이트의 회색 톤과 어울리는 밝은 패널
- 그림자: 너무 강하지 않은 떠 있는 느낌

즉, 화면 전체를 덮는 모달이 아니라
`작은 대화창`처럼 보이는 것이 맞습니다.

### 4-3. 내부 구성

패널 안에는 아래 순서가 적절합니다.

1. 챗봇 헤더
2. 짧은 인사말
3. FAQ 버튼 목록
4. 선택한 질문에 대한 답변 영역
5. 마지막 행동 버튼

가장 중요한 점은, 처음부터 긴 문장을 보여주지 않는 것입니다.
사용자는 “무엇을 눌러야 하는지”가 먼저 보여야 합니다.

## 5. 추천 대화 흐름

### 5-1. 첫 화면

패널이 처음 열리면 아래 정도가 적당합니다.

- 제목: `HAGOBOGO Assistant`
- 인사말: `무엇이 궁금하신가요? 아래에서 자주 묻는 질문을 선택해 주세요.`

그 아래에 질문 버튼 5~10개를 세로 목록으로 둡니다.

### 5-2. 질문 선택 방식

사용자가 질문 버튼을 누르면
채팅처럼 답변이 아래에 추가되거나,
간단히 답변 카드 하나가 열리는 방식이 좋습니다.

이번 1차 구현에서는 너무 복잡한 채팅 로그보다
`질문 선택 -> 답변 표시` 구조가 더 안정적입니다.

### 5-3. 마지막 연결

답변 맨 아래에는 항상 아래 중 하나를 둡니다.

- `Business Inquiries로 문의하기`
- `제안서 보기`
- `다른 질문 보기`

이 중 가장 중요한 것은 `Business Inquiries로 문의하기`입니다.

즉, 챗봇은 끝이 아니라 문의 전환을 위한 중간 단계여야 합니다.

## 6. FAQ 질문 추천안

초기 버전에서는 질문 수를 너무 많이 잡지 않는 것이 좋습니다.
처음에는 아래 7개 정도가 가장 적절합니다.

1. `HAGOBOGO는 어떤 제품인가요?`
2. `누가 주로 사용하는 제품인가요?`
3. `어떤 나라에서 판매되고 있나요?`
4. `제품의 강점은 무엇인가요?`
5. `오프라인 유통에 어떤 장점이 있나요?`
6. `제안서는 어디서 볼 수 있나요?`
7. `비즈니스 문의는 어떻게 하나요?`

이 질문 구성이 좋은 이유는,
사용자가 실제로 물을 가능성이 높은 순서와
대장님이 유도하고 싶은 문의 흐름이 잘 맞기 때문입니다.

## 7. 답변 작성 원칙

답변 문장은 짧고 분명해야 합니다.

이번 FAQ 답변은 아래 원칙을 지키는 것이 좋습니다.

- 한 답변은 2~4문장 정도로 짧게 쓴다.
- 숫자나 성과는 이미 사이트에 있는 표현만 사용한다.
- 의료적 진단처럼 보일 수 있는 표현은 피한다.
- 마지막 문장은 항상 다음 행동으로 연결한다.

예:

- `HAGOBOGO는 남성 생식 건강을 집에서 간편하게 확인하고 관리할 수 있도록 돕는 솔루션입니다. 자세한 도입 상담이 필요하시면 Business Inquiries를 이용해 주세요.`

즉, 챗봇 답변은 “설명”과 “전환”을 동시에 가져가야 합니다.

## 8. 버튼 동작 계획

### 8-1. 챗봇 버튼

- 클릭하면 패널 열기
- 다시 클릭하면 패널 닫기

### 8-2. 패널 닫기

아래 2가지는 기본으로 넣는 것이 좋습니다.

- 우측 상단 닫기 버튼
- 패널 바깥 클릭 시 닫기

### 8-3. 문의 연결 버튼

`Business Inquiries로 문의하기` 버튼을 누르면
기존에 이미 구현된 `Business Inquiries` 모달을 열도록 연결하는 것이 가장 좋습니다.

이 방식이 좋은 이유:

- 새 문의 UI를 또 만들 필요가 없다.
- 기존 문의 저장/메일 전송 흐름을 그대로 쓸 수 있다.
- 사용자가 혼란스럽지 않다.

## 9. 다국어 대응 방향

현재 사이트는 `EN`, `ES`, `FR`, `KR` 언어 전환 구조가 이미 있습니다.

따라서 챗봇도 처음부터 다국어 구조를 염두에 두는 것이 맞습니다.

다만 1차 구현에서는 아래 순서를 권장합니다.

1. 먼저 `EN` 기준 구조를 만든다.
2. 그 다음 기존 번역 파일 구조에 맞춰 `ES`, `FR`, `KR` 문구를 추가한다.

즉, UI 구조는 한 번만 만들고
문구만 번역 데이터로 분리하는 방식이 가장 안전합니다.

## 10. 추천 구현 방식

이번 1차 구현은 아래 구조가 적절합니다.

- `Dashboard` 안에 챗봇 열림 상태 추가
- `ChatbotPanel` 같은 별도 컴포넌트 생성
- 질문/답변 데이터는 번역 파일 또는 별도 상수 파일로 분리
- `Business Inquiries` 버튼과 연결

이 방식의 장점:

- 지금 구조와 잘 맞는다.
- 추후 AI형 챗봇으로 바꿔도 UI 재사용이 가능하다.
- FAQ 데이터 유지보수가 쉽다.

## 11. 예상 사용자 흐름

가장 자연스러운 사용자 흐름은 아래와 같습니다.

1. 사용자가 우하단 챗봇 버튼을 본다.
2. 버튼을 눌러 작은 패널을 연다.
3. 자주 묻는 질문 중 하나를 선택한다.
4. 짧은 답변을 읽는다.
5. 더 자세한 상담이 필요하면 `Business Inquiries`로 넘어간다.

이 흐름은 현재 랜딩 페이지 목적과 잘 맞습니다.

왜냐하면 이 사이트는 “오래 대화하게 만드는 것”보다
“이해시키고 문의로 이어지게 만드는 것”이 더 중요하기 때문입니다.

## 12. 이번 단계에서 넣지 않을 것

기획상 아래 항목은 1차 범위에서 제외하는 것이 좋습니다.

- 자유 입력창
- 생성형 AI 답변
- 채팅 이력 저장
- 관리자 답변 학습
- 외부 메시징 플랫폼 연동

이 항목들은 나중에 2차 확장으로 보는 것이 맞습니다.

## 13. 1차 구현 완료 기준

아래가 되면 1차 챗봇 도입은 성공으로 볼 수 있습니다.

- 우하단 챗봇 버튼 클릭 시 패널이 열린다
- FAQ 질문 5~10개가 보인다
- 질문 선택 시 답변이 보인다
- `Business Inquiries`로 자연스럽게 연결된다
- 현재 사이트의 톤과 어울린다
- 모바일에서도 열고 닫는 데 문제가 없다

## 14. 비판적 관점에서 본 주의사항

이번 챗봇은 “도움이 되는 짧은 안내” 수준을 넘으면 안 됩니다.

특히 조심해야 할 점은 아래입니다.

- 의료 효능처럼 들리는 표현
- 실제 진단을 대체하는 듯한 표현
- 너무 긴 답변
- 문의보다 챗봇 사용 자체에 머무르게 만드는 구조

즉, 잘 만든 챗봇은 오래 대화시키는 챗봇이 아니라
필요한 정보를 짧게 주고 다음 행동으로 보내는 챗봇입니다.

## 15. 최종 제안

아톰 판단으로는,
대장님 사이트의 챗봇 1차 도입은 아래 방식이 가장 적절합니다.

- 현재 챗봇 버튼은 그대로 유지
- 클릭 시 작은 FAQ 패널 열기
- 질문은 7개 안팎으로 제한
- 답변은 짧고 명확하게 구성
- 마지막은 항상 `Business Inquiries` 연결

이 방식은 구현 난이도 대비 효과가 좋고,
운영 리스크도 가장 낮습니다.

대장님, 이 기획안으로 괜찮으시면 다음 단계로는 실제 코드 작성 전에
`FAQ 문구 초안 + 화면 구성안 + 구현 순서`를 더 구체적으로 적은
실행 계획서로 이어가겠습니다.
