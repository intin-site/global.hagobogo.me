# Business Inquiries 팝업 구현 계획서

## 1. 문서 목적

이 문서는 메인 화면의 `Business Inquiries` 버튼을 눌렀을 때 열리는 팝업 UI를 첨부 이미지 기준으로 구현하기 위한 계획서입니다.

이번 계획은 Figma MCP를 사용하지 않고, 대장님이 첨부한 팝업 이미지를 직접 기준으로 삼아 작성했습니다.  
따라서 이 문서는 실제 화면에 최대한 가깝게 구현하기 위한 구체적인 설계 문서 역할을 합니다.

## 2. 이미지 기준 핵심 해석

첨부 이미지를 기준으로 보면 이 팝업은 단순한 폼이 아니라, 기존 랜딩 페이지 위에 떠오르는 `중앙 집중형 문의 모달`입니다.

핵심 특징은 아래와 같습니다.

- 배경 전체가 어둡고 흐리게 처리되어 있다.
- 중앙에 큰 흰색 카드가 뜬다.
- 카드 상단에는 `Business Inquiries` 제목이 중앙 정렬된다.
- 입력 항목은 세로 1열로 정렬되어 있다.
- 필드는 `Name`, `Position`, `Title`, `Company Name`, `Email`, `Inquiry` 순서다.
- `Inquiry`는 일반 입력창이 아니라 큰 텍스트 영역이다.
- 텍스트 영역 우측 상단에는 `0/3000` 글자 수 표시가 있다.
- 하단에는 `Cancel`, `Save` 버튼이 좌우로 나란히 배치되어 있다.

즉, 이 팝업은 "문의자 기본 정보 + 문의 내용"을 한 번에 받는 전형적인 비즈니스 문의 폼입니다.

## 3. 구현 목표

이 팝업 구현의 목표는 아래와 같습니다.

- 현재 랜딩 페이지 톤을 유지하면서도 팝업이 명확히 구분되게 만든다.
- 이미지와 최대한 가까운 배치와 분위기를 재현한다.
- 사용자가 입력 흐름을 헷갈리지 않게 한다.
- 나중에 실제 전송 기능이 붙더라도 구조를 다시 바꾸지 않게 설계한다.

## 4. 전체 레이아웃 계획

### 4-1. 배경 오버레이

팝업이 열리면 기존 페이지 위에 전체 화면 오버레이를 깝니다.

- 위치: `fixed`, 전체 화면
- 배경: 검정 계열 반투명
- 시각 효과: 약한 `blur`
- 목적:
  - 기존 콘텐츠를 뒤로 보내기
  - 팝업에 시선을 집중시키기
  - 현재 페이지 안에서 팝업이 열린 상태라는 것을 분명히 보여주기

이미지상 배경은 완전히 검지 않고, 회색 화면이 어둡게 눌린 정도입니다.  
그래서 구현 시에도 너무 진한 검정보다는 `짙은 회색 + 블러` 조합이 더 맞습니다.

### 4-2. 팝업 카드

팝업 본체는 화면 중앙에 정렬되는 큰 흰색 카드입니다.

- 정렬: 화면 중앙
- 모양: 세로로 긴 직사각형 카드
- 코너 라운드: 매우 크게 적용
- 배경색: 밝은 흰색 계열
- 그림자: 카드 주변이 은은하게 떠 보이는 정도

이미지상 카드는 꽤 큰 편입니다.  
따라서 데스크톱 기준으로는 "작은 모달"이 아니라 "중형 이상 모달"로 잡아야 합니다.

권장 크기 방향:

- 데스크톱 최대 폭: 약 `760px ~ 860px`
- 데스크톱 내부 패딩: 충분히 넓게
- 모바일에서는 `100% - 좌우 여백` 기준으로 축소

## 5. 세부 UI 구조

### 5-1. 제목 영역

카드 상단 중앙에 `Business Inquiries` 제목이 배치됩니다.

구성 방향:

- 중앙 정렬
- 현재 사이트 헤더보다 조금 더 묵직한 타이포 사용
- 색상은 짙은 회색 계열
- 제목 아래 별도 설명 문구는 이미지상 보이지 않으므로 초기 구현에서는 넣지 않는 것이 맞습니다

비판적으로 보면, 보조 설명을 넣으면 친절해질 수는 있지만 이미지와 달라지고 시각 밀도가 올라갑니다.  
따라서 이번 구현은 제목 단독 구성이 더 정확합니다.

### 5-2. 입력 필드 영역

입력 필드는 모두 세로 1열 구조입니다.

필드 순서:

1. Name
2. Position
3. Title
4. Company Name
5. Email
6. Inquiry

각 항목은 아래 구조를 따릅니다.

- 라벨 텍스트
- 입력창

라벨 특징:

- 필드 위쪽
- 좌측 정렬
- 진한 회색보다 조금 연한 회색
- 가독성이 높고 간결한 스타일

입력창 특징:

- 밝은 회색 배경
- 보더가 거의 안 보이거나 매우 옅음
- 코너 라운드가 둥글다
- 높이가 비교적 크다
- 텍스트 입력 영역이 넓고 단정하다

### 5-3. Inquiry 텍스트 영역

`Inquiry`는 다른 필드보다 훨씬 큰 텍스트 영역입니다.

구성 요소:

- 큰 textarea
- 우측 상단의 글자 수 표시 `0/3000`

구현 시 주의점:

- 글자 수 표시는 textarea 바깥이 아니라, 같은 시각 그룹 안에 보여야 합니다
- 사용자가 입력하면 실시간으로 글자 수가 바뀌어야 합니다
- 최대 글자 수는 `3000`으로 제한하는 것이 이미지와 맞습니다

비판적으로 보면, 처음에는 글자 수 표시를 생략하고 싶을 수 있지만, 이미지에서 꽤 분명하게 보이는 요소라 빠지면 완성도가 떨어집니다.

### 5-4. 하단 버튼 영역

하단에는 버튼 두 개가 가로로 배치됩니다.

- 왼쪽: `Cancel`
- 오른쪽: `Save`

버튼 특징:

- 두 버튼의 크기는 거의 동일
- `Cancel`은 흰색 배경 + 진한 테두리
- `Save`는 진한 회색 배경 + 흰색 텍스트
- 두 버튼 모두 라운드가 약하게 적용

이 구조는 역할 분리가 명확합니다.

- `Cancel`: 팝업 닫기
- `Save`: 작성 내용 제출

## 6. 추천 동작 설계

### 6-1. 열기

- 메인 화면의 `Business Inquiries` 버튼 클릭
- 배경 오버레이 표시
- 모달 카드 표시
- 첫 번째 입력 필드인 `Name`에 자동 포커스

### 6-2. 닫기

닫는 방법은 아래 3가지 중 2가지는 반드시 있어야 합니다.

- `Cancel` 버튼 클릭
- `Esc` 키 입력
- 오버레이 클릭

여기서 비판적으로 보면, 오버레이 클릭 닫기는 편하지만 사용자가 긴 문의를 쓰다가 실수로 닫을 위험이 있습니다.  
따라서 초기 구현에서는:

- `Cancel` 버튼
- `Esc` 키

두 개는 넣고, 오버레이 클릭 닫기는 선택적으로 두는 것이 안전합니다.

### 6-3. 저장

`Save` 클릭 시 아래 흐름으로 처리합니다.

1. 필수 입력값 검사
2. 이메일 형식 검사
3. 이상 없으면 저장 또는 제출 처리
4. 완료 메시지 또는 성공 상태 표시

초기 단계에서는 실제 서버 전송이 없어도 됩니다.  
하지만 폼 구조와 검증 흐름은 실제 전송이 붙을 것을 고려해 만들어야 합니다.

## 7. 입력 항목별 역할 정리

각 항목은 의미가 다르므로 단순히 보이는 대로만 구현하면 안 됩니다.

- `Name`: 문의자 이름
- `Position`: 직책
- `Title`: 문의 제목
- `Company Name`: 회사명
- `Email`: 회신용 이메일
- `Inquiry`: 상세 문의 내용

이 구조는 B2B 문의 흐름에 적절합니다.  
특히 `Title`과 `Inquiry`를 분리한 점은 실제 응대 효율에 도움이 됩니다.

## 8. 유효성 검사 계획

최소한 아래 검사는 필요합니다.

- `Name`: 비어 있으면 오류
- `Title`: 비어 있으면 오류
- `Company Name`: 비어 있으면 오류
- `Email`: 비어 있으면 오류, 형식 검사
- `Inquiry`: 비어 있으면 오류

`Position`은 선택 항목으로 둘지 필수 항목으로 둘지 결정이 필요합니다.  
이미지상으로는 동일한 중요도로 보이지만, 실제 사용성 기준으로는 선택 항목이 더 자연스럽습니다.

## 9. 시각 스타일 계획

현재 사이트 톤과 이미지 기준을 함께 보면 아래 방향이 가장 적절합니다.

- 배경 오버레이: 차분한 어두운 회색
- 카드 배경: 밝은 흰색
- 라벨 색상: 중간 회색
- 입력창 배경: 매우 연한 회색
- 주요 버튼 배경: 짙은 회색
- 버튼 텍스트: 흰색 또는 짙은 회색

주의점:

- 현재 메인 사이트가 회색 계열 기반이라 팝업도 같은 계열을 유지해야 자연스럽습니다
- 버튼 색을 과하게 튀게 만들면 랜딩 페이지와 이질감이 생깁니다

## 10. 반응형 계획

이미지는 데스크톱 기준으로 보이지만 실제 구현은 반응형이어야 합니다.

### 데스크톱

- 중앙 카드 폭을 넉넉히 유지
- 입력 필드 간 세로 간격도 충분히 확보
- 하단 버튼은 가로 2열 유지

### 모바일

- 카드 폭을 화면에 맞게 축소
- 내부 패딩을 줄이되 너무 답답하지 않게 유지
- 버튼은 상황에 따라 세로 2열 전환도 허용
- 긴 라벨과 오류 문구가 줄바꿈되어도 깨지지 않게 처리

## 11. 접근성 계획

팝업은 반드시 접근성을 고려해야 합니다.

- `role="dialog"`
- `aria-modal="true"`
- 제목과 모달 연결용 `aria-labelledby`
- 탭 이동이 팝업 안에 머물도록 처리
- `Esc`로 닫기
- 포커스가 팝업 닫힘 후 원래 버튼으로 돌아가도록 처리

이 부분은 겉으로는 보이지 않지만 품질 차이를 크게 만듭니다.

## 12. 구현 파일 구조 제안

현재 프로젝트 구조 기준으로 아래처럼 나누는 것이 적절합니다.

- `src/components/InquiryModal.jsx`
  - 팝업 UI 전체
- `src/components/Dashboard.jsx`
  - `Business Inquiries` 버튼과 팝업 열기 연결
- `src/index.css`
  - 팝업 오버레이, 카드, 입력창, 버튼 스타일

지금 단계에서는 별도 훅 없이 시작해도 충분합니다.  
하지만 실제 제출 로직이 붙으면 `useInquiryForm.js` 또는 `useInquiryModal.js`로 분리할 수 있습니다.

## 13. 단계별 구현 계획

### 1단계. 껍데기 구현

- `Business Inquiries` 버튼 클릭으로 팝업 열기
- 전체 화면 오버레이 추가
- 중앙 카드 배치
- 제목과 하단 버튼 배치

### 2단계. 필드 구현

- 5개의 단일 입력 필드 추가
- 1개의 textarea 추가
- 글자 수 표시 추가

### 3단계. 동작 구현

- 입력 상태 저장
- 글자 수 계산
- `Cancel` 동작
- `Esc` 닫기

### 4단계. 검증 구현

- 필수값 검사
- 이메일 형식 검사
- 오류 메시지 표시

### 5단계. 제출 연결

- 임시 저장 처리 또는 성공 메시지 구현
- 이후 실제 메일/API 연동 구조 연결

## 14. 현재 기준 최종 권장안

현재 첨부 이미지 기준으로 가장 적절한 구현 방향은 아래와 같습니다.

- 팝업은 이미지처럼 큰 중앙 카드 형태로 만든다.
- 제목은 중앙 단독 배치로 유지한다.
- 입력 항목은 `Name`, `Position`, `Title`, `Company Name`, `Email`, `Inquiry` 순서를 그대로 따른다.
- `Inquiry`는 `0/3000` 카운터가 있는 큰 textarea로 구현한다.
- 하단 버튼은 `Cancel`과 `Save` 2개 구조를 유지한다.
- 스타일은 현재 사이트의 회색 계열 분위기를 그대로 이어간다.

## 15. 구현 전 확인이 필요한 항목

실제 코드 작성 전, 아래만 확정하면 구현 품질이 더 좋아집니다.

- `Position`을 필수 입력으로 볼지 선택 입력으로 볼지
- 오버레이 클릭으로 닫기를 허용할지
- `Save` 클릭 시 실제 전송을 할지, 우선은 임시 성공 처리만 할지
- 성공 후 팝업을 닫을지, 완료 메시지를 잠시 보여줄지

이 문서는 첨부 이미지 기준의 실제 구현용 계획서입니다.  
즉시 구현 단계로 넘어가도 무리가 없는 수준으로 작성했습니다.

## 16. 1차 구현 범위

1차 구현은 "실제로 열리고, 입력되고, 닫히고, 기본 검증이 되는 팝업"을 만드는 단계입니다.

포함 범위:

- `Business Inquiries` 버튼 클릭 시 팝업 열기
- 어두운 오버레이와 중앙 카드 표시
- 제목, 입력 필드 6개, `0/3000` 카운터 표시
- `Cancel`, `Save` 버튼 배치
- 기본 입력 상태 관리
- 필수값 검사와 이메일 형식 검사
- `Esc` 키 닫기
- 현재 언어 전환 구조와 연결되는 기본 다국어 처리
- 데스크톱/모바일 기본 반응형 대응

1차 구현에서는 제외하는 것이 맞는 항목:

- 실제 메일 또는 API 전송
- 로딩 스피너와 중복 제출 방지
- 오버레이 클릭 닫기 정책 세밀화
- 성공 메시지 애니메이션
- 포커스 트랩 같은 고급 접근성 보강
- 공통 입력 컴포넌트화

## 17. 2차 고도화 범위

2차 고도화는 1차 구현이 안정적으로 동작한 뒤 품질과 운영성을 높이는 단계입니다.

포함 범위:

- 실제 제출 처리 연결
- 제출 성공/실패 상태 UI
- 로딩 상태 및 버튼 비활성화
- 오버레이 클릭 닫기 정책 결정
- 포커스 복귀 및 포커스 트랩 강화
- 에러 표시 방식 세밀화
