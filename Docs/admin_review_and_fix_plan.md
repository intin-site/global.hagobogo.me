# 관리자 기능 검수 및 수정 계획

## 1. 문서 목적
- 현재 관리자 기능과 관련된 코드, 설정, 배포 구조를 다시 검수해 실제 운영에 맞게 수정하기 위한 기준 문서입니다.
- 초보자도 이해할 수 있도록 `문제 원인 → 실제 위험 → 수정 방향 → 검수 방법` 순서로 정리합니다.
- 이 문서는 아직 코드를 고치는 단계가 아니라, 무엇을 어떤 순서로 고쳐야 하는지 정하는 기획안입니다.

## 2. 현재 상황 한눈에 보기
- 관리자 페이지는 겉으로는 비밀번호가 있는 보호 화면처럼 보이지만, 실제로는 프론트 코드 안에 비밀번호가 들어 있어 보안이 약합니다.
- 관리자 화면에서 바꾸는 뉴스, 판매 수치, Dot_blue 빈도는 서버에 저장되지 않고 현재 브라우저에만 저장됩니다.
- 문의 전송 API와 관리자 설정 API가 같은 환경 변수 이름을 공유하고 있어 배포 설정이 바뀌면 일부 기능이 쉽게 깨질 수 있습니다.
- 린트가 통과하지 않아 현재 코드 상태는 품질 기준이 안정적이지 않습니다.
- 일부 코드는 브라우저 외 환경에서 바로 오류가 날 수 있어 이후 테스트, SSR, 사전 렌더링 확장 시 문제가 될 수 있습니다.

## 3. 핵심 문제 요약

### 3-1. 관리자 인증 구조가 실제 보안 구조가 아님
#### 현재 상태
- 프론트 코드에서 관리자 비밀번호를 직접 비교합니다.
- 로그인 여부도 `sessionStorage`에 저장된 값으로만 판단합니다.
- Apps Script 쪽에도 별도의 관리자 비밀번호 검증이 있어, 프론트와 서버 기준이 이중으로 갈라져 있습니다.

#### 왜 문제인가
- 프론트 번들을 열어보면 비밀번호를 확인할 수 있습니다.
- 브라우저 저장소 값을 수동으로 바꾸면 인증을 우회할 수 있습니다.
- 프론트 비밀번호와 Apps Script 비밀번호가 서로 다르면, 화면은 열리는데 저장 요청은 실패하는 식의 혼란이 생깁니다.

#### 수정 방향
- 관리자 인증 기준은 한 곳으로 통합합니다.
- 권장 방향은 `Apps Script Script Properties`의 `ADMIN_PAGE_PASSWORD`를 단일 진실 원본으로 사용하는 방식입니다.
- 프론트는 비밀번호를 직접 알고 있지 않고, 사용자가 입력한 비밀번호를 서버 검증 요청으로만 확인하도록 바꿉니다.
- 인증 성공 여부도 단순 `sessionStorage true/false`가 아니라, 최소한 서버 검증을 통과한 상태를 기준으로 흐름을 설계합니다.

#### 검수 포인트
- 프론트 코드에 관리자 비밀번호 문자열이 남아 있지 않은지 확인
- 비밀번호가 틀릴 때 관리자 화면 진입과 저장 요청이 모두 막히는지 확인
- Apps Script의 `ADMIN_PAGE_PASSWORD` 값만 바꾸면 전체 관리자 인증 기준이 같이 바뀌는지 확인

### 3-2. 관리자 설정이 전역 운영값이 아니라 개인 브라우저 임시값으로 저장됨
#### 현재 상태
- 뉴스 ticker, 판매 수치, Dot_blue 빈도가 `localStorage`에 저장됩니다.
- 즉 설정을 바꾼 브라우저에서만 값이 보이고, 다른 사람 브라우저나 실제 운영 방문자 화면에는 반영되지 않을 수 있습니다.

#### 왜 문제인가
- 관리자 기능이라는 이름과 실제 동작이 다릅니다.
- 운영자가 값을 바꿨다고 생각했는데 실제 배포 사이트 방문자에게는 반영되지 않을 수 있습니다.
- 기기 교체, 브라우저 캐시 삭제, 다른 PC 사용 시 설정이 사라진 것처럼 보일 수 있습니다.

#### 수정 방향
- 운영에 반영되어야 하는 값은 브라우저 저장소가 아니라 서버 측 저장소로 옮깁니다.
- 현재 구조상 가장 자연스러운 저장 위치는 `Apps Script Script Properties` 또는 Google Sheets의 별도 설정 시트입니다.
- 이번 범위에서는 구조를 단순하게 유지하기 위해 다음처럼 분리하는 안을 권장합니다.

#### 권장 저장 구조
- 알림 수신 메일: `Script Properties`
- 관리자 비밀번호: `Script Properties`
- 판매 수치: `Script Properties`
- Dot_blue 발생 빈도: `Script Properties`
- 뉴스 ticker: 데이터 길이가 길고 언어별 묶음이 있으므로 `Script Properties` 또는 전용 설정 시트 중 하나를 선택

#### 저장소 선택 기준
- 값이 짧고 단순한 설정: `Script Properties`
- 길고 구조가 있는 목록 데이터: 전용 설정 시트가 관리와 백업에 더 유리

#### 검수 포인트
- 관리자 화면에서 저장한 값이 다른 브라우저와 다른 PC에서도 동일하게 보이는지 확인
- 페이지 새로고침 후에도 동일한 값이 유지되는지 확인
- 관리자 변경값이 메인 화면에 실제로 반영되는지 확인

### 3-3. 문의 API와 관리자 API가 서로 다른 계약인데 같은 환경 변수를 사용함
#### 현재 상태
- `VITE_BUSINESS_INQUIRY_API_URL` 하나를 문의 전송과 관리자 설정 요청이 함께 사용합니다.
- 그런데 관리자 요청은 `GET_NOTIFICATION_EMAIL`, `SET_NOTIFICATION_EMAIL` 같은 Apps Script 전용 `action` 값을 기대합니다.
- 반면 Vercel 서버리스 API는 이 관리자 `action`을 처리하지 않습니다.

#### 왜 문제인가
- 배포 환경에서 URL 하나만 바뀌어도 관리자 저장 기능이 갑자기 깨질 수 있습니다.
- 문서와 실제 코드가 서로 다른 구조를 설명하게 되어 운영자가 헷갈리기 쉽습니다.
- 나중에 유지보수할 때 “어느 기능이 어느 서버를 호출하는지” 파악하기 어려워집니다.

#### 수정 방향
- 환경 변수와 API 역할을 명확히 분리합니다.
- 예시:
  - `VITE_BUSINESS_INQUIRY_API_URL`: 문의 접수용
  - `VITE_ADMIN_SETTINGS_API_URL`: 관리자 설정 조회/저장용
- 만약 실제 운영 구조를 Apps Script 하나로 통일할 계획이라면, 문의와 관리자 모두 Apps Script에서 처리하게 만들고 프론트에서도 변수 이름을 역할별로 분리합니다.
- 반대로 Vercel API를 계속 쓸 계획이라면, 관리자용 엔드포인트도 별도로 만들어 계약을 맞춰야 합니다.

#### 검수 포인트
- 문의 전송과 관리자 저장이 각각 올바른 URL로 요청되는지 확인
- 환경 변수를 하나 잘못 바꿨을 때 다른 기능까지 연쇄적으로 망가지지 않는 구조인지 확인
- 운영 문서, `.env.example`, README 설명이 실제 코드와 일치하는지 확인

### 3-4. 린트가 실패하는 상태라 품질 기준이 무너져 있음
#### 현재 상태
- `npm run lint`가 실패합니다.
- 대표 문제:
  - `vite.config.js`의 `__dirname` 사용
  - `DotEngine`, `DotBlueEngine`에서 렌더 중 `performance.now()` 호출
  - `useDotEngine` 내부 콜백 참조 구조 문제
  - `ChatbotPanel`의 effect 내부 동기 `setState`

#### 왜 문제인가
- 지금은 화면이 보여도, 나중에 React 규칙이 더 엄격해지면 오류가 커질 수 있습니다.
- CI 품질 게이트를 붙이기 어렵습니다.
- 팀 기준이 “빌드는 되지만 경고성 오류는 무시” 쪽으로 흐르면 장기적으로 유지보수가 불안정해집니다.

#### 수정 방향
- 린트 통과를 이번 수정 범위의 필수 완료 조건으로 둡니다.
- 각 오류는 단순 무시가 아니라 구조를 바꾸는 방식으로 해결합니다.
- 예를 들어:
  - `__dirname` 문제는 ESM 방식으로 교체
  - 렌더 중 시간 계산은 상태 또는 RAF 업데이트 시점으로 이동
  - Hook 내부 순환 참조는 선언 순서와 콜백 구조를 정리
  - effect 안의 즉시 `setState`는 상태 파생 방식으로 전환

#### 검수 포인트
- `npm run lint`가 오류 0건으로 끝나는지 확인
- 린트만 맞추려고 기능 동작이 바뀌지 않았는지 함께 확인

### 3-5. 브라우저 전용 코드가 초기 상태에서 직접 실행됨
#### 현재 상태
- 일부 상태 초기값이 `window.innerWidth`, `window.innerHeight`를 바로 읽습니다.
- 같은 파일 안에서도 다른 상태는 브라우저 환경 체크가 있는데, 이 부분만 빠져 있습니다.

#### 왜 문제인가
- 지금은 브라우저에서만 실행하므로 바로 안 보일 수 있습니다.
- 하지만 테스트, SSR, 사전 렌더링, 정적 분석 도구 적용 시 즉시 `window is not defined` 오류가 날 수 있습니다.

#### 수정 방향
- 브라우저 객체가 필요한 값은 기본 안전값으로 시작한 뒤 `useEffect` 또는 안전한 초기화 함수에서 계산하도록 통일합니다.
- 같은 파일 안에서 브라우저 접근 규칙을 하나의 패턴으로 맞춥니다.

#### 검수 포인트
- 브라우저 없는 환경에서도 import 단계에서 오류가 나지 않는지 확인
- 첫 렌더 직후 화면이 과도하게 흔들리거나 레이아웃이 깨지지 않는지 확인

## 4. 비밀번호 관리 구조 재정리
### 현재 판단
- 현재 실제 운영 비밀번호는 Apps Script의 `Script Properties`에 있는 `ADMIN_PAGE_PASSWORD`일 가능성이 큽니다.
- 하지만 프론트는 여전히 하드코딩된 비밀번호를 기준으로 관리자 화면 진입을 판단하고 있습니다.
- 즉 “실제 비밀번호 저장 위치”와 “프론트 진입 로직”이 서로 다른 기준으로 움직이고 있습니다.

### 수정 원칙
- 관리자 비밀번호는 코드 안에 두지 않습니다.
- 관리자 비밀번호는 Apps Script `Script Properties`에서만 관리합니다.
- 프론트는 비밀번호 입력값을 직접 비교하지 않고, 서버 검증 결과만 받습니다.
- 비밀번호 변경 기능은 이번 범위에서는 제외하고, 먼저 “현재 비밀번호 기준 통일”을 우선합니다.

### 초보자 기준 설명
- 지금 상태는 자물쇠 비밀번호를 문 밖 안내판에 적어둔 것과 비슷합니다.
- 바르게 고치려면 비밀번호는 문 안쪽 서버에만 있고, 바깥 화면은 “맞는지 확인만 요청”해야 합니다.

- 린트 오류 전체 해결
- 브라우저 전용 초기화 코드 안전화

### 선택 수정
- 관리자 인증 세션 만료 시간 추가
- 관리자 설정 변경 이력 남기기
- 설정 데이터를 시트와 속성 중 어디에 둘지 장기 운영 기준으로 정리

## 7. 상세 검수 시나리오
### 관리자 인증 검수
- 틀린 비밀번호 입력 시 관리자 화면에 들어가지 못하는지 확인
- 맞는 비밀번호 입력 시 관리자 화면에 진입하는지 확인
- 인증 없이 설정 저장 API를 직접 호출하면 거부되는지 확인

### 관리자 설정 검수
- 뉴스 ticker 저장 후 다른 브라우저에서도 같은 값이 보이는지 확인
- 판매 수치 저장 후 메인 화면 수치가 실제로 바뀌는지 확인
- Dot_blue 빈도 저장 후 새로고침해도 유지되는지 확인
- 알림 수신 메일 주소 저장/조회가 정상 동작하는지 확인

### 문의 기능 검수
- 문의 전송이 정상 저장되는지 확인
- 관리자 API 분리 후에도 문의 기능이 깨지지 않는지 확인
- 잘못된 API 주소 설정 시 어떤 기능이 실패하는지 메시지가 명확한지 확인

### 품질 검수
- `npm run lint` 통과
- `npm run build` 통과
- 관리자 화면, 메인 화면, 문의 모달의 주요 흐름 수동 확인

## 10. 최종 판단
- 현재 상태는 “일단 화면은 동작하는 프로토타입”에 가깝고, “운영 관리자 기능”으로 보기에는 구조적 위험이 큽니다.
- 특히 관리자 인증과 설정 저장 구조는 가장 먼저 바로잡아야 합니다.
- 이번 수정은 단순 버그 수정이 아니라, 운영 기준에 맞게 관리자 기능의 책임과 저장 위치를 다시 세우는 작업으로 보는 것이 맞습니다.

## 11. 코드 수정 전 확인 사항

- 문의 기능도 Apps Script로 계속 운영할지, Vercel API 구조를 유지할지 결정
