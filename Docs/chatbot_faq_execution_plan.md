# HAGOBOGO FAQ형 챗봇 실행 계획서

## 1. 문서 목적

이 문서는 `FAQ형 챗봇 도입 기획안`을 실제 구현 단계로 옮기기 전에,
무엇을 어떤 순서로 만들지 정리한 실행 계획서입니다.

즉, 이번 문서는 아래 3가지를 분명하게 만드는 것이 목적입니다.

- 실제로 화면에 어떤 FAQ를 넣을지
- 챗봇 패널이 어떤 구조로 동작할지
- 어떤 파일을 어떻게 수정할지

이번 문서는 아직 코드 구현 문서는 아닙니다.
하지만 이 문서까지 정리되면 바로 개발에 들어갈 수 있는 수준을 목표로 합니다.

## 2. 이번 구현의 범위

이번 1차 구현 범위는 아래로 제한합니다.

1. 기존 우하단 챗봇 버튼 유지
2. 클릭 시 FAQ 패널 열기
3. 질문 7개 제공
4. 질문 클릭 시 답변 표시
5. `Business Inquiries` 연결 버튼 제공
6. 다국어 구조 고려

이번 단계에서 제외하는 항목은 아래와 같습니다.

- 자유 입력형 채팅
- 생성형 AI 답변
- 서버 저장형 대화 기록
- 실시간 상담원 연결

즉, 이번 버전은 “정보 안내 + 문의 전환”까지만 책임집니다.

## 3. 핵심 UX 목표

이번 챗봇은 아래 사용자 경험을 만들면 성공입니다.

1. 사용자가 챗봇 버튼을 눌렀을 때 부담 없이 패널이 열린다.
2. 복잡한 입력 없이 질문 버튼만 눌러도 답을 볼 수 있다.
3. 답변을 읽은 뒤 자연스럽게 `Business Inquiries`로 넘어갈 수 있다.

쉽게 말하면,
이번 챗봇은 “사이트 안의 작은 안내 데스크”처럼 동작해야 합니다.

## 4. 화면 구조 상세안

### 4-1. 패널 위치

- 위치: 우하단 챗봇 버튼 위
- 정렬: 오른쪽 기준 정렬
- 레이어: 기존 콘텐츠보다 위
- 성격: 모달이 아니라 작은 패널

이렇게 해야 현재 페이지 흐름을 깨지 않고
사용자가 “필요할 때만 잠깐 보는 도구”로 받아들입니다.

### 4-2. 패널 내부 구성

패널은 아래 순서로 구성합니다.

1. 헤더
2. 짧은 안내 문구
3. FAQ 질문 리스트
4. 선택된 답변 카드
5. 행동 버튼 영역

### 4-3. 헤더 구성

헤더에는 아래 요소를 넣습니다.

- 제목: `HAGOBOGO Assistant`
- 닫기 버튼: `X`

이 헤더는 너무 화려할 필요는 없습니다.
오히려 현재 사이트 톤처럼 절제된 느낌이 더 적절합니다.

### 4-4. 안내 문구

초기 안내 문구는 아래 방향이 좋습니다.

- EN: `What would you like to know? Choose a frequently asked question below.`
- KR 기준 의미: `무엇이 궁금하신가요? 아래 자주 묻는 질문을 선택해 주세요.`

이 문구는 짧고 바로 행동을 유도해야 합니다.

## 5. FAQ 질문 초안

초기 질문은 아래 7개를 권장합니다.

### 5-1. 질문 1

`HAGOBOGO는 어떤 제품인가요?`

답변 초안:

`HAGOBOGO는 남성 생식 건강을 집에서 간편하게 확인하고 관리할 수 있도록 돕는 솔루션입니다. 제품과 활용 방식이 더 궁금하시면 제안서와 비즈니스 문의를 통해 자세히 안내받으실 수 있습니다.`

### 5-2. 질문 2

`누가 주로 사용하는 제품인가요?`

답변 초안:

`임신을 준비하는 커플, 자기 건강 관리에 관심 있는 사용자, 남성 난임 예방과 조기 확인에 관심 있는 소비자가 주요 사용자층입니다. 실제 유통 관점에서는 커플 키트 제안도 가능합니다.`

### 5-3. 질문 3

`어떤 나라에서 판매되고 있나요?`

답변 초안:

`HAGOBOGO는 한국과 일본에서 성과를 만들었고, 미국·캐나다·싱가포르 등 여러 국가로 확장 중입니다. 글로벌 유통 협업이 가능한 구조라는 점이 강점입니다.`

### 5-4. 질문 4

`제품의 강점은 무엇인가요?`

답변 초안:

`사용자가 스마트폰으로 직접 확인할 수 있는 직관성, 앱 기반 기록 관리, 그리고 글로벌 인증 기반 신뢰성이 핵심 강점입니다. 단순 검사 도구가 아니라 지속 관리 경험까지 함께 제공합니다.`

### 5-5. 질문 5

`오프라인 유통에 어떤 장점이 있나요?`

답변 초안:

`여성용 제품 중심으로 구성된 가족계획 카테고리를 확장해 새로운 고객층과 추가 객단가를 만들 수 있다는 점이 장점입니다. 커플 번들 구성과 골든존 배치 전략도 함께 제안할 수 있습니다.`

### 5-6. 질문 6

`제안서는 어디서 볼 수 있나요?`

답변 초안:

`사이트 안 제안서 영역에서 바로 확인할 수 있고, 언어별 PDF 다운로드도 가능합니다. 필요하면 담당자에게 직접 제안 자료를 요청하실 수도 있습니다.`

### 5-7. 질문 7

`비즈니스 문의는 어떻게 하나요?`

답변 초안:

`챗봇 아래의 Business Inquiries 연결 버튼을 누르면 바로 문의 모달이 열립니다. 회사명, 이메일, 문의 내용을 남기시면 확인 후 회신받을 수 있습니다.`

## 6. 답변 작성 원칙

답변은 아래 기준으로 고정하는 것이 좋습니다.

- 2문장 이상, 4문장 이하
- 과장 표현 금지
- 의료 진단처럼 보이는 문장 금지
- 숫자는 기존 사이트 안에서 이미 쓰는 수치만 사용
- 마지막은 다음 행동을 유도

즉, 이번 챗봇 답변은 “세일즈 문구”가 아니라
`짧고 신뢰감 있는 안내`에 가까워야 합니다.

## 7. 행동 버튼 구조

답변 아래 행동 버튼은 아래 2개를 권장합니다.

1. `Business Inquiries`
2. `View Proposal`

여기서 우선순위는 `Business Inquiries`가 더 높아야 합니다.

이유:

- 실제 사이트 목표가 문의 전환이기 때문입니다.
- 제안서 보기만 하고 끝나는 흐름보다 문의 연결이 더 중요합니다.

## 8. 패널 동작 규칙

### 8-1. 열기

- 챗봇 버튼 클릭 시 패널 오픈

### 8-2. 닫기

- 닫기 버튼 클릭
- 패널 바깥 클릭
- `Esc` 키

### 8-3. 질문 선택

- 질문 버튼 클릭 시 해당 질문이 활성화
- 활성 질문에 대한 답변 카드가 아래에 표시
- 다른 질문 클릭 시 답변 카드 내용만 교체

이번 1차 버전에서는
채팅 로그가 계속 누적되는 방식보다
`한 번에 하나의 답변 카드`가 더 적절합니다.

이유:

- 구현이 단순하다
- 모바일에서 더 읽기 쉽다
- FAQ 성격에 맞다

## 9. 다국어 대응 계획

현재 사이트 언어 구조는 `EN`, `ES`, `FR`, `KR`입니다.

이번 챗봇도 아래 방식으로 가는 것이 맞습니다.

- 질문/답변 데이터는 언어별 번역 데이터 안에 둔다
- UI 구조는 공통 컴포넌트로 한 번만 만든다

즉, 코드 구조는 하나고
문구만 언어별로 갈아끼우는 방식입니다.

## 10. 추천 파일 구조

구현 시 손댈 가능성이 높은 파일은 아래와 같습니다.

- `src/components/Dashboard.jsx`
- `src/components/ChatbotPanel.jsx` 신규 추가
- `src/index.css`
- `src/i18n/translations.js`

각 파일의 역할:

- `Dashboard.jsx`
  - 챗봇 열림 상태 관리
  - 챗봇 버튼 클릭 처리
  - `Business Inquiries` 모달 연결

- `ChatbotPanel.jsx`
  - FAQ 패널 UI 렌더링
  - 질문 리스트 표시
  - 답변 카드 표시
  - 닫기 동작 처리

- `src/index.css`
  - 챗봇 패널 레이아웃
  - 모바일 대응
  - 버튼/카드 스타일

- `translations.js`
  - 언어별 챗봇 제목, 안내 문구, 질문, 답변, 버튼 텍스트 저장

## 11. 구현 순서 제안

개발 순서는 아래가 가장 안전합니다.

1. `ChatbotPanel` 컴포넌트 뼈대 만들기
2. `Dashboard`에 열기/닫기 상태 연결
3. 정적 FAQ 1개로 UI 먼저 확인
4. FAQ 7개 구조로 확장
5. `Business Inquiries` 연결 버튼 붙이기
6. 다국어 문구 연결
7. 모바일 레이아웃 점검

이 순서가 좋은 이유는
복잡한 문구 작업 전에 화면 구조가 맞는지 먼저 확인할 수 있기 때문입니다.

## 12. 시각 디자인 방향

현재 사이트에 맞는 챗봇 패널은 아래 느낌이 적절합니다.

- 색상: 회색 기반, 밝은 카드, 시안 포인트 최소 사용
- 폰트: 현재 사이트와 동일한 Barlow Condensed / Pretendard 계열 유지
- 버튼: 너무 장난스럽지 않고 깔끔한 라운드 버튼
- 답변 카드: 카드 한 장이 떠 있는 듯한 정돈된 구성

즉, “메신저 앱” 느낌보다
`브랜드 안내 카드` 느낌이 더 맞습니다.

## 13. 예상 리스크와 대응

### 13-1. 질문이 너무 많아지는 문제

질문이 많아지면 챗봇이 아니라 메뉴판이 됩니다.

대응:

- 1차는 7개로 고정
- 추가는 운영 후 판단

### 13-2. 답변이 길어지는 문제

답변이 길면 사용자가 읽지 않습니다.

대응:

- 4문장 이하 원칙 유지
- 필요하면 `Business Inquiries`로 넘김

### 13-3. 의료적 표현 리스크

의료기기/난임 관련 맥락에서는 표현이 민감합니다.

대응:

- 진단/치료 보장처럼 보이는 문장 금지
- “확인”, “관리”, “도움”, “안내” 중심 표현 사용

## 14. 구현 완료 기준

아래 조건이 만족되면 실제 구현 시작 전 준비가 끝난 것으로 봅니다.

- FAQ 질문 7개와 답변 초안 확정
- 버튼 동작 구조 확정
- `Business Inquiries` 연결 방식 확정
- 다국어 문구 저장 위치 확정
- 수정 파일 범위 확정

## 15. 최종 제안

아톰 판단으로는,
이번 챗봇 1차 구현은 아래 수준이 가장 적절합니다.

- 우하단 버튼 유지
- 우하단 소형 패널 오픈
- 질문 7개
- 한 번에 하나의 답변 카드 표시
- 마지막 행동은 `Business Inquiries` 중심

이 방식이면 구현 난이도는 낮고,
사이트 목적과도 가장 잘 맞습니다.

대장님, 이 실행 계획서가 괜찮으시면
다음 단계는 실제 코드 작성입니다.
여기서부터는 제가 구현에 들어가기 전에 다시 한 번 확인받고 진행하겠습니다.
