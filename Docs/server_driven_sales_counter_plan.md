# 서버 기준 판매 카운트 및 실시간 애니메이션 기획안

## 1. 문서 목적
- 이 문서는 판매 카운트를 브라우저가 아니라 서버 기준으로 관리하도록 구조를 바꾸기 위한 기획안입니다.
- 목표는 `어느 브라우저에서 열어도 같은 카운트 값이 보이게 하는 것`과 `웹페이지가 켜져 있을 때만 파란 점 애니메이션이 자연스럽게 보이게 하는 것`입니다.
- 이번 문서는 대장님이 확정한 조건인 `시간 규칙 기반 자동 증가`, `Admin의 제품 판매 빈도 설정(Dot_blue 발생 빈도) 연동`, `60초~100초 주기 폴링`을 기준으로 정리합니다.

## 2. 현재 문제
### 지금 구조
- 현재 판매 카운트는 화면 안에서 파란 점이 충돌할 때 브라우저가 직접 숫자를 올립니다.
- 즉, 카운트를 만드는 주체가 서버가 아니라 브라우저입니다.

### 왜 문제가 되는가
- 웹페이지를 닫으면 브라우저가 멈추므로 카운트도 더 이상 늘지 않습니다.
- 어떤 브라우저는 켜져 있고, 어떤 브라우저는 꺼져 있으면 서로 다른 숫자가 보일 수 있습니다.
- 새 브라우저에서 접속했을 때 기존 브라우저와 다른 숫자가 보일 수 있습니다.

## 3. 목표
- 판매 카운트의 진실 원본은 서버 하나만 사용합니다.
- 서버는 시간 규칙에 따라 자동으로 카운트를 누적합니다.
- 자동 증가 속도는 관리자 페이지의 `제품 판매 빈도 설정 (Dot_blue 발생 빈도)` 값을 기준으로 계산합니다.
- 웹페이지가 켜져 있는 브라우저는 서버 숫자 변화를 감지했을 때만 파란 점 애니메이션을 재생합니다.
- 애니메이션이 끝난 뒤 숫자를 1씩 따라 올립니다.
- 관리자 페이지에서 카운트를 재설정하면, 그 값이 새로운 공식 기준값이 됩니다.

## 4. 핵심 아이디어
### 역할 분리
- 서버:
  - 실제 판매 카운트 값을 보관합니다.
  - 시간 규칙에 따라 자동 증가를 계산합니다.
  - 관리자 재설정을 처리합니다.
- 브라우저:
  - 서버의 최신 카운트 값을 읽습니다.
  - 값이 늘어난 것을 감지하면 애니메이션을 재생합니다.
  - 애니메이션이 끝난 뒤 화면 숫자를 따라 올립니다.

### 쉬운 비유
- 서버는 `공식 전광판 관리실`입니다.
- 브라우저는 `전광판 화면`입니다.
- 숫자를 바꾸는 권한은 관리실에만 있고, 화면은 관리실이 바꾼 숫자를 보고 따라 움직입니다.

## 5. 추천 구조
### 추천 방향
- 서버에 `currentSalesCount` 하나를 공식 판매 수치로 둡니다.
- 서버는 `Dot_blue 발생 빈도` 설정값을 기준으로 자동 증가 속도를 계산합니다.
- 브라우저는 `60초~100초` 주기로 서버 값을 읽어옵니다.
- 브라우저가 켜져 있는 동안 서버 값이 증가한 것이 보이면 파란 점 애니메이션을 재생하고, 끝난 뒤 숫자를 올립니다.

### 왜 이 방향이 좋은가
1. 모든 브라우저가 같은 서버 값을 기준으로 보게 됩니다.
2. 웹페이지가 꺼져 있어도 서버 기준 숫자는 계속 누적될 수 있습니다.
3. 관리자 재설정도 서버 한 곳만 바꾸면 되므로 기준이 단순합니다.
4. 브라우저 애니메이션은 연출만 담당하므로 구조가 더 명확해집니다.

## 6. 서버 데이터 구조 제안
### Script Properties 기준 예시
- `SITE_CURRENT_SALES_COUNT`
  - 현재 공식 판매 카운트
- `SITE_SALES_UPDATED_AT`
  - 마지막 카운트 변경 시각
- `SITE_LAST_SALES_TICK_AT`
  - 마지막 자동 증가 계산 기준 시각
- `SITE_DOT_BLUE_RANGE_JSON`
  - 관리자 페이지에서 설정한 자동 증가 기준 범위
- `SITE_SETTINGS_UPDATED_AT`
  - 관리자 설정 변경 시각

### 관리자 재설정 시
- 관리자 페이지에서 `108500`으로 저장하면
- 서버의 `SITE_CURRENT_SALES_COUNT = 108500`
- 서버의 `SITE_LAST_SALES_TICK_AT = 저장 시각`
- 이후 자동 증가 규칙에 따라 다시 `108501`, `108502`, `108503` 순서로 누적됩니다.

## 7. 서버 API 구조 제안
### 7-1. 공개 조회 API
- 브라우저가 로그인 없이 읽을 수 있어야 합니다.
- 조회 시점에 서버는 먼저 자동 증가 누적분을 반영한 뒤 최신값을 반환하는 것이 좋습니다.
- 예시 응답:

```json
{
  "ok": true,
  "siteSettings": {
    "currentSalesCount": 108524,
    "salesUpdatedAt": "2026-03-14T11:20:00.000Z",
    "lastSalesTickAt": "2026-03-14T11:20:00.000Z",
    "dotBlueSpawnFrequencyRange": {
      "min": 900,
      "max": 1440
    }
  }
}
```

### 7-2. 자동 증가 계산 로직
- 이번 구조에서는 브라우저가 `+1`을 보내지 않습니다.
- 서버가 현재 시각과 마지막 계산 시각을 비교해, 늘어나야 할 개수를 스스로 계산합니다.
- 예시 동작:
  - 현재 값 읽기
  - 마지막 계산 시각 읽기
  - `Dot_blue 발생 빈도 설정`을 기준으로 증가 개수 계산
  - 계산된 수만큼 `currentSalesCount` 반영
  - 마지막 계산 시각 갱신
  - 변경 시각 갱신

### 7-3. 관리자 재설정 API
- 관리자 인증 후 새 숫자를 저장합니다.
- 저장 시 기존 카운트를 새 숫자로 덮어씁니다.
- 동시에 자동 증가 기준 시각도 `현재 시각`으로 다시 맞춥니다.

## 8. Dot_blue 발생 빈도 설정을 자동 증가에 연결하는 방식
### 의미 해석
- 현재 관리자 설정의 `Dot_blue 발생 빈도`는 이미 `시간당 예상 발생 수`처럼 사용되고 있습니다.
- 따라서 이 값을 자동 증가 규칙에 그대로 연결하는 것이 가장 자연스럽습니다.

### 추천 계산 방식
- `min`과 `max`의 평균값을 사용합니다.
- 공식:
  - `hourlyRate = (min + max) / 2`
- 예:
  - `min = 900`
  - `max = 1440`
  - 평균 = `1170`
  - 시간당 약 `1170` 증가

### 왜 평균값을 추천하는가
1. 계산이 단순합니다.
2. 운영자가 결과를 예측하기 쉽습니다.
3. 재설정 후 흐름이 안정적입니다.
4. 테스트와 검수가 쉽습니다.

### 대안
- 랜덤 방식으로도 만들 수 있지만, 초기에 검증이 어려워질 수 있습니다.
- 1차 구현은 평균값 방식이 더 안전합니다.

## 9. 브라우저 동작 방식
### 첫 진입 시
1. 서버에서 `currentSalesCount`를 읽습니다.
2. 브라우저 화면은 이 값을 그대로 표시합니다.
3. 처음 들어올 때는 과거 증가분 애니메이션을 전부 재생하지 않습니다.

### 웹페이지가 열려 있는 동안
1. 브라우저가 `60초~100초` 간격으로 서버 값을 다시 조회합니다.
2. 새 서버 값이 현재 화면 값보다 크면 차이를 계산합니다.
3. 차이가 `1`이면:
   - 파란 점 애니메이션 1회 재생
   - 애니메이션 종료 후 숫자 `+1`
4. 차이가 `2 이상`이면:
   - 차이만큼 순서대로 애니메이션을 재생할지
   - 일부만 재생하고 숫자는 빠르게 따라잡을지 정책을 정해야 합니다.

## 10. 애니메이션 정책 추천
### 추천 정책
- 차이가 `1~3`이면 개수만큼 파란 점 애니메이션을 순서대로 재생합니다.
- 차이가 `4 이상`이면 모든 개수를 다 재생하지 말고 아래처럼 처리합니다.

### 추천 처리 방식
- 최근 1~2건만 대표 애니메이션으로 보여줍니다.
- 숫자는 최종 서버 값까지 빠르게 따라갑니다.

### 왜 이런 제한이 필요한가
- 폴링 간격이 `60초~100초`이면 한 번 조회할 때 여러 건 차이가 날 수 있습니다.
- 파란 점을 수십 번 연속으로 보여주는 것은 오히려 어색할 수 있습니다.
- 사용자는 최신값을 빠르게 보는 것이 더 중요할 수 있습니다.

## 11. Apps Script 기준 현실적인 구현 판단
### 가능 범위
- Apps Script는 현재 값 저장, 재설정, 조회 API 역할에는 잘 맞습니다.
- `LockService`를 써서 동시에 여러 요청이 와도 숫자가 꼬이지 않게 보호할 수 있습니다.
- Apps Script가 호출될 때마다 "지금 시각 기준으로 누적분 재계산"을 하는 구조도 구현 가능합니다.

### 한계
- Apps Script는 실시간 서버 푸시(WebSocket, SSE) 용도로는 적합하지 않습니다.
- Apps Script는 아무 호출도 없는데 혼자 계속 실행되는 서버는 아닙니다.

### 결론
- 현재 프로젝트에서는 `Apps Script + 시간 기준 누적 계산 + 60초~100초 주기 폴링` 구조가 가장 현실적입니다.

## 12. 시간 트리거 사용 판단
### 왜 필요한가
- 브라우저가 아무도 열지 않은 시간에도 서버 숫자가 실제로 미리 반영되어 있기를 원할 수 있습니다.
- 이 경우 Apps Script `시간 기반 트리거`를 함께 쓰는 것이 더 안정적입니다.

### 추천
- `공개 조회 시점 재계산`만으로도 최신값 계산은 가능합니다.
- 하지만 대장님 요구사항에 더 가깝게 가려면 `시간 트리거 + 조회 시점 재계산`을 같이 쓰는 것이 좋습니다.

### 추천 이유
1. 브라우저가 없어도 서버 값이 주기적으로 정리됩니다.
2. 조회 시점에도 한 번 더 보정할 수 있어 누락에 강합니다.
3. 여러 브라우저가 동시에 열려도 기준값이 더 안정적입니다.

## 13. 추천 통신 방식
### 추천: 주기적 폴링
- 브라우저는 `60초~100초` 간격으로 최신 카운트를 읽습니다.
- 구현이 단순하고 유지보수가 쉽습니다.
- Apps Script 환경과도 잘 맞습니다.
- 호출 부담도 과하지 않게 유지할 수 있습니다.

### 비추천: WebSocket/SSE
- 현재 구조와 맞지 않고 구현 비용이 커집니다.
- Apps Script와 직접 연결하기도 까다롭습니다.

## 14. 관리자 재설정 규칙
### 추천 규칙
- 관리자 페이지에서 입력한 값은 `현재 공식 카운트`를 직접 덮어씁니다.
- 저장 시 자동 증가 기준 시각도 함께 현재 시각으로 재설정합니다.
- 예:
  - 현재 서버 값 `108524`
  - 관리자가 `108300` 저장
  - 서버 공식값은 즉시 `108300`
  - 이후 증가 시 `108301`, `108302`로 누적

### 장점
- 계산 규칙이 단순합니다.
- 기준값과 누적 증가분을 따로 생각하지 않아도 됩니다.
- 운영자가 숫자를 수동 보정하기 쉽습니다.
- 재설정 직후 다시 예전 증가분이 섞여 들어오는 문제를 막을 수 있습니다.

## 15. 구현 시 주의점
### 주의점 1. 서버 값이 먼저여야 함
- 브라우저가 먼저 숫자를 올리고 나중에 서버에 맞추는 구조는 다시 값이 어긋날 수 있습니다.
- 반드시 서버 값이 먼저 바뀌고, 브라우저는 이를 따라가야 합니다.

### 주의점 2. 폴링 주기는 길지만 애니메이션 정책이 필요함
- 이번 요구사항은 `60초~100초` 폴링이므로, 한 번 조회할 때 차이가 여러 건일 가능성이 큽니다.
- 따라서 늘어난 개수 전부를 애니메이션할지 제한해야 합니다.

### 주의점 3. 차이값 처리 정책이 필요함
- 페이지가 꺼져 있던 동안 서버 값이 많이 늘었으면
- 애니메이션을 몇 번까지 재생할지 명확한 기준이 있어야 합니다.

### 주의점 4. 숫자 점프가 어색하지 않게 해야 함
- 차이가 클 때는 애니메이션보다 최신값 반영을 우선해야 합니다.

### 주의점 5. Apps Script는 상시 실행 서버가 아님
- 엄밀히 말하면 "가만히 있어도 무조건 계속 도는 서버"로 이해하면 안 됩니다.
- 실제로는 조회 시점이나 시간 트리거 실행 시점에 누적 계산이 반영됩니다.

## 16. 추천 구현 순서
1. Apps Script의 공식 판매 카운트 키를 `currentSalesCount` 중심 구조로 단순화합니다.
2. `Dot_blue 발생 빈도 설정`을 기준으로 자동 증가량을 계산하는 함수를 추가합니다.
3. 마지막 자동 증가 반영 시각(`lastSalesTickAt`) 저장 구조를 추가합니다.
4. 공개 조회 API가 호출될 때 현재 시각 기준 누적분을 먼저 계산한 뒤 최신 카운트를 반환하도록 바꿉니다.
5. 관리자 저장 API가 공식 카운트와 기준 시각을 함께 재설정하도록 바꿉니다.
6. Apps Script 시간 트리거를 추가해 브라우저가 없어도 주기적으로 서버 값이 반영되게 합니다.
7. 프론트는 `60초~100초` 주기 폴링으로 서버 최신값을 읽도록 바꿉니다.
8. 서버 값 증가 감지 시 파란 점 애니메이션 후 숫자를 올리는 큐 구조를 만듭니다.
9. 차이값이 큰 경우 애니메이션 제한 정책을 넣습니다.
10. 여러 브라우저에서 동시에 열어 같은 숫자가 보이는지 검수합니다.

## 17. 검수 시나리오
1. 브라우저 A와 브라우저 B를 동시에 열었을 때 같은 숫자로 시작하는지 확인
2. 서버 기준 시간이 지나 카운트가 증가하면 두 브라우저 모두 애니메이션 뒤 숫자가 증가하는지 확인
3. 브라우저를 껐다가 다시 켰을 때 최신 서버 값으로 바로 시작하는지 확인
4. 관리자에서 판매 수치를 재설정하면 모든 브라우저가 새 값 기준으로 맞춰지는지 확인
5. 카운트 차이가 여러 건 발생했을 때 애니메이션이 과도하지 않은지 확인
6. `Dot_blue 발생 빈도` 값을 바꾼 뒤 다음 누적 속도가 새 기준으로 적용되는지 확인

## 18. 최종 판단
- 대장님이 제안하신 방향은 현재 문제를 해결하는 데 가장 올바른 방향입니다.
- 특히 `서버가 숫자를 만들고, 브라우저는 그 숫자 변화를 보기 좋게 표현한다`는 기준이 중요합니다.
- 이렇게 바꾸면 어느 브라우저에서 열어도 같은 숫자를 볼 수 있고, 관리자가 값을 재설정해도 기준이 흔들리지 않습니다.
- 이번에 확정된 `시간 규칙 기반 자동 증가 + Dot_blue 발생 빈도 설정 연동 + 60초~100초 폴링` 조합은 현재 Apps Script 구조에서도 충분히 현실적인 방향입니다.

## 19. 코드 수정 전 확인 요청
- 다음 단계로 실제 코드 수정에 들어가려면 아래 기준으로 진행하는 것이 좋습니다.

1. `Dot_blue 발생 빈도 min/max`는 평균값으로 계산
2. Apps Script는 `시간 트리거 + 조회 시점 재계산`을 함께 사용
3. 브라우저는 `60초~100초` 랜덤 폴링
4. 차이값이 클 때는 대표 애니메이션 1~2회만 재생
