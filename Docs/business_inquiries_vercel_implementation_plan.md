# Business Inquiries Vercel 서버리스 1차 구현 계획서

## 1. 문서 목적

이 문서는 `Business Inquiries` 팝업의 메일 발송 기능을
`Vercel 서버리스 함수 + Resend` 구조로 실제 구현하기 위한 1차 작업 계획서입니다.

이번 문서는 추상 설계가 아니라, 바로 구현에 착수할 수 있도록 아래 항목을 기준으로 정리합니다.

- 어떤 파일을 만들지
- 어떤 순서로 작업할지
- 각 단계에서 무엇을 검증할지
- 어디서 문제가 생기기 쉬운지

## 2. 1차 구현 목표

1차 구현의 목표는 아래 한 가지입니다.

`Business Inquiries` 팝업에서 입력한 내용을 실제로 관리자 이메일 주소로 전송할 수 있는 상태 만들기

즉, 이번 단계에서는 아래까지만 달성하면 충분합니다.

- 팝업 `Save` 버튼과 API 연결
- Vercel 서버리스 함수 구현
- Resend 메일 발송 연결
- 성공/실패 메시지 표시
- 최소 검증 적용

## 3. 1차 구현 범위

포함:

- 프론트 `fetch` 호출 추가
- 서버리스 함수 1개 추가
- Resend SDK 또는 HTTP API 연동
- 환경 변수 사용
- 필수 필드 검증
- 이메일 형식 검증
- 문의 길이 검증
- 성공/실패 응답 표시

제외:

- 데이터베이스 저장
- 관리자 대시보드
- 스팸 필터 고도화
- CAPTCHA
- 자동 응답 메일
- 문의 내역 이력 저장
- rate limiting 고도화

## 4. 권장 파일 구조

현재 프로젝트 기준으로 아래 구조를 권장합니다.

```text
site_hagobogo/

      InquiryModal.jsx
    i18n/
      translations.js
    lib/
      inquiryApi.js

  api/

  Docs/

```

## 5. 파일별 역할

### 5-1. `src/components/InquiryModal.jsx`

역할:

- 폼 입력값 관리
- `Save` 클릭 시 제출 처리
- 제출 중 상태 반영
- 성공/실패 결과 표시

이번 단계에서 수정할 항목:

- 기존 `Save`의 임시 닫기 동작 제거
- API 호출 연결
- 제출 성공 시 사용자 안내
- 제출 실패 시 사용자 안내

### 5-2. `src/lib/inquiryApi.js`

역할:

- 프론트에서 서버리스 API를 호출하는 공통 함수

이 파일을 두는 이유:

- API 호출 코드를 컴포넌트에서 분리할 수 있다.
- 나중에 엔드포인트가 바뀌어도 수정 지점을 줄일 수 있다.
- 테스트와 유지보수가 쉬워진다.

### 5-3. `api/business-inquiry.js`

역할:

- Vercel 서버리스 함수
- 메일 발송의 실제 백엔드 처리 담당

주요 책임:

- `POST` 요청 수신
- 입력값 검증
- Resend 호출
- 응답 반환

### 5-4. `src/i18n/translations.js`

역할:

- 성공/실패 메시지 번역 관리
- 필요 시 전송 중 문구 관리

## 6. 구현 단계

### 1단계. 프론트 제출 구조 정리

목표:

- `InquiryModal` 내부에서 제출 흐름을 명확하게 만든다.

작업:

- `isSubmitting` 상태 추가
- `submitError` 상태 추가
- `submitSuccess` 상태 추가
- `Save` 클릭 시 API 호출 함수 실행 구조 준비

검증:

- 클릭 시 중복 제출이 막히는지 확인
- 기존 유효성 검사가 유지되는지 확인

### 2단계. API 호출 유틸 작성

목표:

- 프론트에서 서버리스 함수를 호출하는 코드를 분리한다.

파일:

- `src/lib/inquiryApi.js`

권장 함수 예시:

- `submitBusinessInquiry(payload)`

역할:

- `fetch` 호출
- JSON 응답 파싱
- 실패 시 에러 throw

검증:

- 성공 응답과 실패 응답을 구분할 수 있는지 확인

### 3단계. 서버리스 함수 생성

목표:

- `api/business-inquiry.js`를 추가한다.

작업:

- `POST` 외 요청 차단
- 본문 JSON 파싱
- 필수값 검증
- 이메일 형식 검증
- 글자 수 검증
- Resend 호출 준비

검증:

- 잘못된 요청에 `400` 또는 `405`가 반환되는지 확인
- 필수값 누락 시 올바른 메시지가 오는지 확인

### 4단계. Resend 연동

목표:

- 서버리스 함수 안에서 메일 발송이 되도록 한다.

준비물:

- `RESEND_API_KEY`
- `BUSINESS_INQUIRY_TO_EMAIL`
- `BUSINESS_INQUIRY_FROM_EMAIL`

작업:

- 환경 변수 읽기
- 메일 제목 작성
- 메일 본문 작성
- Resend 호출

검증:

- 실제 수신 이메일로 메일이 도착하는지 확인
- 실패 시 서버 오류 응답이 정상 반환되는지 확인

### 5단계. 프론트 성공/실패 메시지 연결

목표:

- 사용자에게 제출 결과를 분명하게 보여준다.

작업:

- 성공 시 안내 문구 표시
- 실패 시 오류 문구 표시
- 성공 시 팝업 자동 닫기 여부 결정

권장 1차안:

- 성공 시 `문의가 정상적으로 접수되었습니다.` 표시
- 짧은 시간 뒤 닫기 또는 버튼 상태 복귀

검증:

- 성공/실패가 화면에서 명확히 구분되는지 확인

### 6단계. 운영 환경 연결

목표:

- GitHub Pages 프론트와 Vercel API를 실제로 연결한다.

작업:

- 운영용 API 주소 확정
- 프론트에서 해당 주소 호출
- CORS 허용 Origin 설정

검증:

- 실제 GitHub Pages 도메인에서 제출이 되는지 확인

## 7. 서버리스 함수 검증 기준

1차 구현에서 최소한 아래 검증은 반드시 포함해야 합니다.

- `name` 비어 있지 않은가
- `companyName` 비어 있지 않은가
- `email` 비어 있지 않은가
- `email` 형식이 맞는가
- `inquiry` 비어 있지 않은가
- `inquiry.length <= 3000` 인가

현재 정책 기준:

- `jobTitle`은 선택값

## 8. 메일 제목/본문 권장 포맷

### 제목

```text
[HAGOBOGO] New Business Inquiry from {companyName}
```

### 본문

```text
Name: {name}
Job Title: {jobTitle}
Company Name: {companyName}
Email: {email}
Language: {language}

Inquiry:
{inquiry}
```

초기에는 가장 읽기 쉬운 plain text 기준이 적절합니다.

## 9. 프론트 제출 상태 설계

`InquiryModal` 안에서 아래 상태를 두는 것이 적절합니다.

- `isSubmitting`
  - 전송 중인지 표시
- `submitError`
  - 서버 오류 메시지 표시
- `submitSuccess`
  - 성공 메시지 표시

버튼 동작 원칙:

- 전송 중에는 `Save` 비활성화
- 전송 중에는 버튼 문구를 `Saving...` 계열로 바꾸는 것도 가능

## 10. 번역 추가 항목

`translations.js`에 아래 문구를 추가하는 것이 좋습니다.

- 제출 성공 문구
- 제출 실패 문구
- 전송 중 문구
- 서버 오류 기본 문구

이유:

- 현재 팝업은 다국어 구조를 이미 가지고 있으므로, 제출 결과 문구만 영어로 남기면 UI 일관성이 깨집니다.

## 11. Vercel 배포 준비 체크리스트

1. Vercel 프로젝트 생성
2. 저장소 연결 여부 결정
3. `api/business-inquiry.js` 함수 배포 확인
4. 환경 변수 등록
5. Resend 도메인 인증 완료
6. 발신 이메일 주소 확인
7. 수신 이메일 주소 확인

## 12. 운영 테스트 체크리스트

1. 정상 입력 시 메일 도착 여부
2. `name` 누락 시 서버 차단 여부
3. 이메일 형식 오류 시 서버 차단 여부
4. 문의 길이 초과 시 차단 여부
5. GitHub Pages에서 실제로 호출되는지 확인
6. 전송 실패 시 프론트 오류 메시지 노출 확인

## 13. 가장 많이 막히는 지점

### 13-1. GitHub Pages와 Vercel의 도메인 분리

프론트와 API 도메인이 다르기 때문에 CORS 처리가 빠지면 운영에서 막힙니다.

### 13-2. Resend 발신 주소 미인증

발신 도메인 인증이 안 되면 메일이 정상 발송되지 않을 수 있습니다.

### 13-3. 프론트에 키를 넣으려는 실수

Resend API 키는 반드시 서버리스 함수 환경 변수에만 있어야 합니다.

### 13-4. 성공과 실패 UI를 너무 늦게 붙이는 문제

메일은 실제로 실패할 수 있기 때문에, 사용자에게 명확한 성공/실패 상태를 보여주는 UI를 초기에 함께 붙여야 합니다.

## 14. 1차 구현 완료 기준

아래 항목이 모두 되면 1차 구현 완료로 봅니다.

- 팝업 `Save` 클릭 시 서버리스 API 호출
- 서버리스 함수가 검증 후 Resend 메일 발송
- 성공 시 사용자 안내 문구 노출
- 실패 시 사용자 오류 문구 노출
- GitHub Pages 운영 화면에서 실제 제출 동작 확인

## 15. 권장 다음 단계

1차 구현이 끝나면 다음 순서로 가는 것이 적절합니다.

1. rate limiting 추가
2. 관리자용 메일 템플릿 개선
3. 자동 응답 메일 검토
4. 제출 로그 저장 여부 검토

즉, 지금은 먼저 `작동하는 최소 메일 발송 구조`를 만드는 것이 우선입니다.
